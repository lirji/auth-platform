## 1. 平台是什么 / 你要接什么

统一权限平台把「身份」和「授权」拆成两块独立能力：

| 能力 | 由谁提供 | 你接它得到什么 |
|---|---|---|
| **身份 / 登录 / SSO** | **Casdoor**（OIDC 提供方，`:8000`） | 一个可验签的 **access token（JWT）**，里面带稳定 userId、租户、用户组 |
| 细粒度授权（ReBAC） | SpiceDB + `auth-platform-server`（`:8200`） | 「主体能否对某资源做某操作」的判定（本手册不展开，见 §8） |

```
                    ┌──────── 授权码 + PKCE ────────┐
   浏览器 SPA ──────▶│          Casdoor (:8000)      │  ← 登录页/会话/SSO
                    └──── access_token (JWT/RS256) ─┘
        │                          ▲
        │  Authorization: Bearer <access_token>
        ▼                          │ JWKS 验签(/.well-known/jwks)
   你的后端服务 (OIDC Resource Server) ── 校验通过 → 拿到 sub/owner/groups
```

**接入统一登录 = 两件事**：

1. **前端**：把「自己的登录页 + 自己发 token」换成 **跳转 Casdoor 登录 → 拿 Casdoor access_token**（授权码 + PKCE）。
2. **后端**：把「自己校验自己的 token」换成 **把 Casdoor 当 OIDC 提供方，用它的 JWKS 验签 access_token**，从 claim 里取用户身份。

> 关键收益：`Casdoor sub` 成为**全局稳定的用户 ID**，所有服务、所有判权都对齐到它。不再各服务各有一套 user 表。

## 2. 核心概念与 Token Claim 契约（先读这张表）

Casdoor 签发的是 **RS256 JWT**。以下 claim 契约来自本仓库真实实测（`docs/plans/.../PHASE0-RESEARCH.md` P0-1），**接入方一律以这张表取值**：

| 语义 | claim | 实测形态 | 用法 |
|---|---|---|---|
| **用户唯一 ID** | `sub` | 稳定 UUID，如 `e56eeac5-101d-…` | **全局主键**：写日志、判权、关联业务数据都用它。**不要用 username** |
| **租户 ID** | `owner` | Casdoor 组织名，如 `built-in` | 多租户隔离键。**只信 token 里的 `owner`，绝不接受前端 header/query 覆盖** |
| **用户组** | `groups` | 字符串数组，元素为**全路径** `<org>/<group>`，如 `built-in/authz-admin` | 归一化（取最后一段）后当粗粒度角色/权限 |
| **角色** | `roles` | 字符串数组（可能为空） | 备用；平台建模主要走 `groups`/`permissions` |
| **业务权限（scope）** | `permissions` | 对象数组，含 `name` | **只有先在 Casdoor 建模并 seed 后才有**（见 §8）。默认可能为空 |
| 签发方 | `iss` | `http://localhost:8000`（dev） | 后端**必须校验** |
| 受众 | `aud` | = 应用的 `client_id` | 后端**建议校验**（防跨应用 token 混用） |
| 标准 OIDC scope | `scope` | 仅 `openid profile …`，**不含业务 scope** | 别指望这里有 `ingest/chat` 之类业务能力 |

> ⚠️ **最容易踩的两点**：① `sub` 是 UUID 不是用户名；② 业务能力**不在** `scope` 里，而在 `groups`（归一化后）或 `permissions`（需先 seed）。详见 §7。

## 3. 接入前准备：在 Casdoor 注册应用

联系平台管理员（或自行登录 Casdoor 后台 `http://localhost:8000`，dev 默认 `admin/123`），为你的项目建一个 **Application**，拿到 `client_id`（前端 PKCE 无需 secret；后端机器身份需要 secret）。

需要向管理员确认/提供：

- **应用名 / client_id**（后端校验 `aud` 时要用）。
- **Redirect URLs（回调地址）**：填你前端的回调页，如 `http://localhost:5273/callback`、生产 `https://yourapp.example.com/callback`。**必须精确登记**，否则授权码回跳被拒。
- **授权模式**：前端 SPA → `Authorization Code`（PKCE）；机器身份 → `Client Credentials`。
- **Scopes**：至少 `openid profile`；要 refresh_token 续期则加 `offline_access`。
- **Token 格式**：`JWT`（RS256）。
- **（可选）用户组 / 角色分配**：如需按组做粗粒度门禁，请管理员把用户加入相应 `group`（决定 token 的 `groups` claim）。

> 平台里已有两个现成应用可复用：管控台前端 `auth-console`（授权码 + PKCE）、机器身份 `app-built-in`（脚本换 token）。新项目建议**独立建应用**，便于按 `aud` 隔离与吊销。
>
> **dev / 多租户快速开通（方案 C：Shared Application）**：`deploy/casdoor-tenant-provision.sh` 一条龙开租户，
> 省去手点后台。每个**项目 = 一租户 = 一 Casdoor org**（Casdoor 是跨项目共享的公共 IAM）。脚本**不再每租户建
> app**：幂等确保 built-in 拥有的 Shared Application（`rag-shared`，`isShared=true`、`orgChoiceMode=Input`）存在，
> 之后只建 org + user（+ 设密码 + 可选写 SpiceDB 成员组）。登录/换 token 用**派生 client_id `<base>-org-<tenant>`**
> （共用同一 shared secret），Casdoor 签出 `aud=[<base>-org-<tenant>]`、`owner=<tenant>`。消费方（如 edge 网关）
> 的 `aud` allowlist 只需一次性登记 base client_id，按 `<base>-org-*` 家族 +（owner,aud）绑定放行——**新增租户
> 零改动、无需重启**。
>
> 多租户 SaaS 型项目推荐走上面的 Shared Application；单租户/独立项目仍可独立建应用（见上文），按 `aud` 隔离与吊销。

## 4. 场景 A：前端 SPA 接入（授权码 + PKCE）

**标准做法**：`authorization_code + PKCE`，**不在前端存任何 secret**。参考实现见 `auth-console/src/auth/oidcConfig.ts`（React + `oidc-client-ts` + `react-oidc-context`）。其它框架（Vue/Angular）用各自的 `oidc-client-ts` 封装同理。

### 4.1 安装

```bash
pnpm add oidc-client-ts react-oidc-context   # React；Vue 用 vue3-oidc 或直接裸用 oidc-client-ts
```

### 4.2 OIDC 配置（抄自 auth-console，已跑通）

```ts
// oidcConfig.ts
import { UserManager, WebStorageStateStore, type UserManagerSettings } from 'oidc-client-ts'

export const oidcSettings: UserManagerSettings = {
  authority: import.meta.env.VITE_CASDOOR_AUTHORITY ?? 'http://localhost:8000', // = issuer
  client_id: import.meta.env.VITE_CASDOOR_CLIENT_ID,        // 你的应用 client_id
  redirect_uri: `${window.location.origin}/callback`,       // 须与 Casdoor 登记一致
  post_logout_redirect_uri: `${window.location.origin}/`,
  response_type: 'code',                                     // 授权码 + PKCE（库自动做 PKCE）
  scope: 'openid profile',                                   // 需续期再加 ' offline_access'
  loadUserInfo: false,                                       // groups 已在 access_token 里，免一次 userinfo/CORS
  automaticSilentRenew: true,
  userStore: new WebStorageStateStore({ store: window.sessionStorage }), // 关标签页即清
}

export const userManager = new UserManager(oidcSettings)
```

> `authority` 会自动拉 `/.well-known/openid-configuration`，authorize/token/jwks/logout 端点全由 discovery 提供，**无需硬编码**。

### 4.3 登录 / 回调 / 登出流程

- **未登录** → `userManager.signinRedirect({ state: { returnTo: location.pathname } })` 跳 Casdoor 登录页。
- **回调页 `/callback`** → `userManager.signinRedirectCallback()`，成功后清掉 URL 上的 `code/state`（`history.replaceState`）再跳回 `returnTo`。
- **登出** → `userManager.signoutRedirect()`。

### 4.4 给业务请求带 token（axios 拦截器，抄自 `api/client.ts`）

```ts
apiClient.interceptors.request.use(async (cfg) => {
  const user = await userManager.getUser()
  if (user && !user.expired && user.access_token) {
    cfg.headers.Authorization = `Bearer ${user.access_token}`
  }
  return cfg
})

// 401 → 静默续期一次（单飞防惊群）→ 仍失败落交互式登录
```

> 参考实现里 401 会先 `signinSilent()` 重试一次，再失败才 `signinRedirect`，避免 token 刚过期就把用户踢去登录页。生产强烈建议照抄这段。

### 4.5 前端读 groups（可选，仅用于 UI 显隐）

```ts
// 解 access_token payload，取归一化后的 groups（'built-in/authz-admin' → 'authz-admin'）
groupsFromToken(user.access_token) // 见 auth-console/src/auth/oidcConfig.ts
```

> **前端解 token 只能用于 UI 显隐（能不能看到某按钮），绝不能当安全边界**——真正的门禁一律在后端按验签后的 claim 判。

## 5. 场景 B：后端服务接入（Spring Boot OIDC Resource Server）

你的服务**不做登录**，只做一件事：**把收到的 `Authorization: Bearer <access_token>` 用 Casdoor 的 JWKS 验签**，验过就信任 claim 里的身份。**范本文件**：`auth-platform-admin/.../SecurityConfig.java`（已加固、curl 全矩阵验过），直接照抄改包名即可。

### 5.1 依赖

```xml
<dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-oauth2-resource-server</artifactId>
</dependency>
<!-- WebFlux 网关（如 Spring Cloud Gateway）则用 ReactiveJwtDecoder，见 §5.5 -->
```

### 5.2 配置（application.yml）

```yaml
server:
  # ⚠️ 必配：Casdoor access_token 常达 8-9KB（内嵌完整 user 对象），超过默认 8KB header 上限会被
  # Tomcat 在鉴权前直接 400（连 /actuator/health 都 400）。Spring Boot 3 的属性名是下面这个
  # （2.x 的 server.max-http-header-size 已失效，配了不生效）。实测本平台 token = 8835B。
  max-http-request-header-size: 64KB

authz:
  security:
    jwk-set-uri: ${CASDOOR_JWKS:http://localhost:8000/.well-known/jwks}
    issuer:      ${CASDOOR_ISSUER:http://localhost:8000}
    client-id:   ${CASDOOR_CLIENT_ID:}   # = 你的应用 client_id，用于 aud 校验；留空则跳过 aud 校验
```

> **这是最隐蔽的坑**：不配 `max-http-request-header-size`，合法 token 会拿到 `400 Bad Request`（不是 401），
> 且看不出跟 header 大小有关——因为 Tomcat 在 Spring Security 之前就拒了。本平台 `auth-platform-admin`
> 已加此配置（见其 `application.yml`），接入方**务必照做**。

### 5.3 JwtDecoder：JWKS 验签 + issuer + （可选）aud（抄自 SecurityConfig）

```java
@Bean
public JwtDecoder jwtDecoder(AdminSecurityProperties props) {
    NimbusJwtDecoder decoder = NimbusJwtDecoder.withJwkSetUri(props.getJwkSetUri()).build();
    List<OAuth2TokenValidator<Jwt>> validators = new ArrayList<>();
    validators.add(new JwtTimestampValidator());              // exp/nbf（含默认 clock skew）
    validators.add(new JwtIssuerValidator(props.getIssuer())); // iss 必须匹配
    if (props.getClientId() != null && !props.getClientId().isBlank()) {
        String aud = props.getClientId();
        validators.add(jwt -> jwt.getAudience().contains(aud)
                ? OAuth2TokenValidatorResult.success()
                : OAuth2TokenValidatorResult.failure(new OAuth2Error("invalid_token", "aud 需含 " + aud, null)));
    }
    decoder.setJwtValidator(new DelegatingOAuth2TokenValidator<>(validators));
    return decoder;
}
```

> **为什么用 `NimbusJwtDecoder.withJwkSetUri` 而不是手解 JWKS**：它自动缓存 + 轮转多把公钥（Casdoor 可能轮换签名 key），手解必踩 key rotation 的坑。

### 5.4 把 groups 映射成 Spring Security 权限（抄自 SecurityConfig）

Casdoor 的 `groups` 是全路径 `built-in/authz-admin`，归一化（取最后一段）成 `authz-admin` 当权限：

```java
@Bean
public JwtAuthenticationConverter jwtAuthenticationConverter() {
    JwtAuthenticationConverter converter = new JwtAuthenticationConverter();
    converter.setJwtGrantedAuthoritiesConverter(jwt -> {
        Collection<GrantedAuthority> authorities = new ArrayList<>();
        Object groups = jwt.getClaim("groups");
        if (groups instanceof Collection<?> col) {
            for (Object g : col) {
                String s = String.valueOf(g);
                int i = s.lastIndexOf('/');
                authorities.add(new SimpleGrantedAuthority(i >= 0 ? s.substring(i + 1) : s));
            }
        }
        return authorities;
    });
    return converter;
}
```

> 想用业务 scope 做门禁？把上面 `jwt.getClaim("groups")` 换/加成解析 `permissions`（对象数组，取每个 `name`）——前提是已按 §8 在 Casdoor seed 过。

### 5.5 SecurityFilterChain（无状态 + 端点授权，抄自 SecurityConfig）

```java
http
  .csrf(AbstractHttpConfigurer::disable)
  .sessionManagement(s -> s.sessionCreationPolicy(SessionCreationPolicy.STATELESS)) // 无 session
  .authorizeHttpRequests(auth -> auth
      .requestMatchers("/actuator/health", "/actuator/info").permitAll()  // 放行探活
      .requestMatchers(HttpMethod.POST, "/your/admin/**").hasAuthority("authz-admin")
      .anyRequest().authenticated())
  .oauth2ResourceServer(o -> o.jwt(j -> j.jwtAuthenticationConverter(jwtAuthenticationConverter)));
```

**取当前用户**：`((Jwt) authentication.getPrincipal()).getSubject()` 即 `sub`；`getClaimAsString("owner")` 即租户。

> **WebFlux 网关（Spring Cloud Gateway）** 用响应式版：`NimbusReactiveJwtDecoder.withJwkSetUri(jwks)` + 同样的 issuer/aud validator。若要在自定义 `GlobalFilter` 里验签换发内部 token（灰度期与旧 token 并存），只注入 `ReactiveJwtDecoder` 组件、**不启用整条 `SecurityWebFilterChain`**，避免与现有 filter 链冲突（设计见 `docs/plans/.../MILESTONE-B-STEP3-CASDOOR-SSO-DESIGN.md`）。

## 6. 场景 C：机器对机器 / 脚本换 Token

后台任务、服务间调用、CI 脚本没有浏览器，用 **password grant（dev 联调）** 或 **client_credentials（生产机器身份）** 直接换 token：

```bash
# password grant（dev 快速拿一个真实 token 做验证；抄自 deploy/casdoor-seed.sh）
curl -s -X POST "http://localhost:8000/api/login/oauth/access_token" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=password&username=admin&password=123&client_id=$CID&client_secret=$CSEC&scope=openid" \
  | jq -r .access_token
```

- **client_credentials**：`grant_type=client_credentials&client_id=…&client_secret=…`，token 的 `sub` 代表机器身份。
- 拿到的 token 用法与用户 token 一致：`Authorization: Bearer <token>` 打你的后端，后端同样 JWKS 验签。
- Casdoor discovery 实测支持 `client_credentials` / `token-exchange` / `password` grant（PHASE0 P0-3）。

> **secret 只存后端/CI 密文**（环境变量 / secret manager），绝不进前端、不进 git。

## 7. 注意事项与常见坑（重点）

| # | 坑 | 正确做法 |
|---|---|---|
| 1 | **用 username 当用户主键** | 一律用 `sub`（UUID，全局稳定）。username 可改、可跨租户重名。历史数据迁移需做 `username → sub` 的一次性 crosswalk（P0-4） |
| 2 | **信任前端传的 tenant** | `tenantId` **只**取已验签 token 的 `owner` claim；**拒绝**任何 header/query 覆盖，否则越权（D5） |
| 3 | **以为 `scope` 里有业务能力** | Casdoor 的 `scope` 只有标准 OIDC scope。业务能力在 `groups`（归一化）或 `permissions`（需先 seed），见 §8 |
| 4 | **groups 没归一化** | `groups` 是全路径 `built-in/authz-admin`；判权前取最后一段 `authz-admin`（`lastIndexOf('/')`） |
| 5 | **手动解析 JWKS** | 用 `NimbusJwtDecoder.withJwkSetUri`（自动缓存 + key 轮转）。手解遇到 Casdoor 换 key 会突然全体 401 |
| 6 | **不校验 iss / aud** | 必校 `iss`；建议校 `aud`（= 你的 client_id），否则别的应用的 token 也能打进来 |
| 7 | **claim 缺失时放行** | `sub`/`owner` 缺失一律 **fail-closed（拒绝）**，不要「读不到就当匿名放过」 |
| 8 | **seed 顺序错，permissions 为空** | Casdoor 的 `permission→role→user` 展开在 **permission 写入时**定死。**必须先给用户分配角色（role.users）再跑 `casdoor-seed.sh`**；反了要重跑脚本让 permission 重建（见脚本头注释） |
| 9 | **redirect_uri 没登记 / 不精确** | 回调地址必须与 Casdoor 应用里登记的**完全一致**（协议+域名+端口+路径），否则授权码回跳被拒 |
| 10 | **token 过期直接踢登录** | 前端 401 先 `signinSilent()` 续期重试一次再交互式登录；要 refresh_token 则 scope 加 `offline_access` |
| 11 | **dev 配置进生产** | dev 的 `runmode=dev`、`admin/123`、`http://localhost:8000`、`socks5Proxy` 等**仅本地**。生产换 HTTPS issuer、真实凭据、去代理，见 §10 |
| 12 | **写后立即判权读不到** | 这是 SpiceDB 一致性语义（ZedToken），属判权侧问题，不是登录问题；见 §8 与 `CLAUDE.md` 一致性档位 |
| 13 | **合法 token 却 400（不是 401）** | Casdoor token ~8.8KB > Tomcat 默认 8KB header 上限，鉴权前就被 400。**必配 `server.max-http-request-header-size: 64KB`**（Spring Boot 3 属性名；2.x 的 `max-http-header-size` 已失效）。见 §5.2 |

## 8. 延伸：从登录到判权

登录只解决「**你是谁**」（authentication）。「**你能对哪个资源做什么**」（authorization）是平台的另一半：

### 8.1 两层门禁

| 粒度 | 依据 | 在哪判 |
|---|---|---|
| **粗粒度**（能不能进这个模块 / 有没有 admin 角色） | token 里的 `groups` / `permissions` | **你自己的后端**，用 §5.4/5.5 的 authority 直接判（`hasAuthority(...)`） |
| **细粒度**（这个用户能不能看**这一篇**文档 / 跨科室数据） | SpiceDB 关系元组（ReBAC） | 调 **`auth-platform-server` 判权服务** |

### 8.2 粗粒度业务 scope：Casdoor 建模 + seed

若你要 `ingest/chat/analytics` 这类业务能力门禁，用 `deploy/casdoor-seed.sh` 把「角色→scope」初始化进 Casdoor（每个 scope 一个 permission，`permission.roles` = 拥有该 scope 的角色）。之后 token 的 `permissions` claim 直接带用户的业务 scope。**5 角色 → scope 映射**（`casdoor-seed.sh`）：

| 角色 | scopes |
|---|---|
| viewer | chat |
| editor | chat, ingest |
| analyst | chat, analytics |
| approver | chat, approve |
| admin | chat, ingest, approve, agent, channel, eval, vision, voice, analytics, role-admin, public-ingest |

⚠️ 见 §7 坑 #8 的 seed 顺序。

### 8.3 细粒度 ReBAC：接判权 SDK（`auth-platform-sdk`）

引入 Spring Boot Starter，声明式判权：

```xml
<dependency>
  <groupId>com.lrj.authz</groupId>
  <artifactId>auth-platform-sdk</artifactId>
</dependency>
```

```yaml
authz:
  client:
    enabled: true
    server-url: http://localhost:8200          # auth-platform-server
    token: ${AUTHZ_CLIENT_TOKEN:}              # server 开了 security 时的 service credential
```

```java
// 1) 提供一个 SubjectResolver bean：从安全上下文取当前用户（就是上面验签出的 Casdoor sub）
@Bean
SubjectResolver subjectResolver() {
    return () -> SubjectRef.user(currentCasdoorSub());   // 例：从 SecurityContext 的 Jwt.getSubject()
}

// 2) 业务方法上声明式判权：从名为 docId 的参数取资源 id，判 document:docId 的 view 权限
@CheckAccess(permission = "view", resourceType = "document", resourceIdParam = "docId")
public Document getDocument(String docId) { ... }
```

判权不通过抛 `AccessDeniedException`。撤权/敏感变更要立即生效时，`@CheckAccess(..., fullyConsistent = true)`。
完整链路与一致性（ZedToken）语义见仓库根 `CLAUDE.md`。

> **身份对齐是前提**：`SubjectResolver` 返回的必须是 **Casdoor `sub`**，与 SpiceDB 里关系元组的 subject 一致，否则判权「空转」（写的是 `user:<sub>`，判的却是别的 id）。这也是为什么先接登录、再接判权。

## 9. 验证方式（分层冒烟）

**逐层验证，每层过了再上一层**——避免「前端登录转圈」时分不清是 Casdoor、后端验签、还是 CORS 的问题。

### Layer 0 — Casdoor 可达、discovery/JWKS 正常

```bash
curl -s http://localhost:8000/.well-known/openid-configuration | jq '{issuer, jwks_uri, authorization_endpoint, token_endpoint}'
curl -s http://localhost:8000/.well-known/jwks | jq '.keys | length'   # 应 ≥ 1
```

### Layer 1 — 能换到一个真实 token，且 claim 契约符合 §2

```bash
AT=$(curl -s -X POST http://localhost:8000/api/login/oauth/access_token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=password&username=admin&password=123&client_id=$CID&client_secret=$CSEC&scope=openid" | jq -r .access_token)

# 解 payload（不验签，只看 claim 形状）
echo "$AT" | cut -d. -f2 | base64 -d 2>/dev/null | jq '{sub, owner, groups, roles, aud, iss}'
# 断言：sub 是 UUID、owner=built-in、groups 形如 ["built-in/xxx"]
```

### Layer 2 — 后端验签生效（401/200 矩阵）

> 前提：后端已按 §5.2 配 `max-http-request-header-size`，否则大 token 一律 400（见下方“大 token 门禁”）。

```bash
# 0) 大 token 门禁：把大 token 打到 permitAll 的 health。200=header 放得下且验签通过；400=没配 header-size
curl -s -o /dev/null -w "%{http_code}\n" -H "Authorization: Bearer $AT" http://localhost:8201/actuator/health  # 期望 200
# 1) 无 token → 401
curl -s -o /dev/null -w "%{http_code}\n" http://localhost:8201/admin/schema                                    # 期望 401
# 2) 篡改签名（翻转末字符，长度不变）→ 401（验签拦下）
curl -s -o /dev/null -w "%{http_code}\n" -H "Authorization: Bearer ${AT%?}A" http://localhost:8201/admin/schema # 期望 401
# 3) 合法 token → 200（有权）或 403（验签过但组不够）——只要非 401 即接入成功
curl -s -o /dev/null -w "%{http_code}\n" -H "Authorization: Bearer $AT" http://localhost:8201/admin/schema      # 期望 200/403
```

> **一键脚本**：以上 Layer 0-2 已封装为 **`deploy/sso-smoke.sh`**（风格同 `spicedb/server/his-smoke.sh`），
> 缺凭据/后端的层自动 SKIP：
> ```bash
> bash deploy/sso-smoke.sh                                                   # 仅 Layer 0
> CASDOOR_CLIENT_ID=… CASDOOR_CLIENT_SECRET=… bash deploy/sso-smoke.sh        # 全量 Layer 0-2
> # 换目标后端：AUTHZ_ADMIN=http://localhost:8300 ADMIN_SMOKE_PATH=/your/protected …
> ```
> 接入自己的服务时，把 `AUTHZ_ADMIN`/`ADMIN_SMOKE_PATH` 指向你的资源服务即可复用同一套断言。

### Layer 3 — 前端端到端

1. 未登录访问受保护页 → 自动跳 Casdoor 登录页。
2. 登录成功 → 回跳 `/callback` → 跳回原页面，且业务请求头带上 `Authorization: Bearer`。
3. 手动等到 token 过期（或调短）→ 请求应静默续期、不把用户踢回登录页。
4. 登出 → 会话清空，再访问受保护页又跳登录。

## 10. 生产化 Checklist

- [ ] **HTTPS issuer**：Casdoor 换成 `https://sso.yourcorp.com`；`iss`/`jwk-set-uri`/前端 `authority` 同步换，去掉 `localhost`。
- [ ] **`runmode = prod`**、关 `isDemoMode`、改掉默认 `admin/123`、去掉 dev 的 `socks5Proxy`（`deploy/casdoor/app.conf` 是 dev 配置）。
- [ ] **放大 header 上限**：`server.max-http-request-header-size: 64KB`（Casdoor token ~9KB > 默认 8KB，否则合法 token 400）。
- [ ] **每应用独立 `client_id`**，后端**开启 `aud` 校验**（`authz.security.client-id` 非空）。
- [ ] **redirect_uri 精确登记**生产域名；前端 `redirect_uri` 用 `window.location.origin` 自适应但仍要登记。
- [ ] **JWKS 高可用**：Casdoor-only 阶段没有 legacy 兜底，SSO 挂 = 全线登录不了；JWKS 需可达 + decoder 缓存兜短时抖动。
- [ ] **secret 进密文存储**（env / secret manager），不进 git、不进前端 bundle。
- [ ] **判权服务写端加固**：`auth-platform-server` 生产必须 `authz.server.security.enabled=true` + 配 token，否则 `/v1/relationships` 写端裸奔（任何可达 :8200 者皆可改授权）。SDK 侧配 `authz.client.token` 对应。
- [ ] **fail-closed 复核**：`sub`/`owner` 缺失、验签失败、JWKS 不可达 → 一律拒绝，不降级放行。
- [ ] **时钟同步**：resource server 与 Casdoor 时钟偏差过大会导致 `exp/nbf` 校验误杀（`JwtTimestampValidator` 有默认容忍窗，但别依赖大偏差）。
- [ ] **历史身份 crosswalk**：老系统若以 username 存关系/数据，上线前做 `username → sub` 迁移（P0-4）。

## 11. 排错 FAQ

| 现象 | 多半原因 | 排查 |
|---|---|---|
| 前端登录后一直转圈 / 回不到应用 | `redirect_uri` 未登记或不精确；`/callback` 未处理 `signinRedirectCallback()` | 对比 Casdoor 应用登记的 Redirect URLs 与实际 `redirect_uri` 是否字符级一致 |
| 合法 token 却 **400 Bad Request**（连 `/actuator/health` 带 token 也 400） | Casdoor token(~9KB) 超 Tomcat 默认 8KB header 上限，鉴权前被拒 | 配 `server.max-http-request-header-size: 64KB`（§5.2）。用 `deploy/sso-smoke.sh` 的“大 token 门禁”定位 |
| 后端一律 401（连合法 token 也 401） | `jwk-set-uri` 不可达 / `issuer` 不匹配（尾斜杠、http vs https） | Layer 0 先确认 JWKS 可达；`iss` 与 `JwtIssuerValidator` 配的完全相等 |
| 合法 token 但 403（登录 OK、权限不足） | `groups` 没配 / 没归一化 / 端点要求的 authority 不匹配 | 解 token 看 `groups`；确认 §5.4 归一化后与 `hasAuthority(...)` 对得上 |
| `permissions` claim 为空，业务 scope 门禁全失效 | 没跑 `casdoor-seed.sh`，或 seed 前没先分配 role.users | 按 §7 坑 #8：先分配角色→再 seed；必要时重跑脚本 |
| 篡改/过期 token 却没被拦 | 后端没真正走 resource-server 验签（可能只解了 payload 没验签） | 确认用 `NimbusJwtDecoder` + validators，而非自己 `base64` 解 payload 就信 |
| CORS 报错（浏览器） | Casdoor / 后端未放行前端 origin | 检查 Casdoor 应用 origin 配置与后端 CORS；`loadUserInfo:false` 可省一次 userinfo 的 CORS |
| 判权总是拒绝（登录正常） | `SubjectResolver` 返回的不是 Casdoor `sub`（对不上 SpiceDB 里的 subject） | 见 §8.3：主体必须是 `sub` |

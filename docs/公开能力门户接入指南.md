# 公开能力门户接入指南

## 1. 定位与边界

`project-portal` 是独立的匿名静态站，用来展示当前公开提供的项目与能力。访问者浏览门户不需要 Casdoor 会话；点击项目后，浏览器进入目标项目自己的登录入口，由目标项目生成 OAuth state、PKCE verifier/challenge，再跳转 Casdoor。

```text
浏览器 → project-portal（匿名静态资源 + 同源 catalog.json）
       → 目标项目 /login（租户选择/确认）
       → Casdoor authorize（目标项目生成 PKCE）
       → 目标项目 callback → 原目标页
```

门户不接收 OIDC callback，不保存 token，不调用 auth-platform 后端、Casdoor API 或目标业务 API，也不替代 `auth-console`。项目登录后的 tenant、audience、scope 和业务授权仍由目标项目、网关、Casdoor 与 auth-platform 校验。

## 2. 本地运行

```bash
# 只启动公开门户（http://localhost:5274）
./dev.sh up --no-infra --no-backend --no-frontend

# 或直接在工程内启动
cd project-portal
corepack pnpm install
corepack pnpm dev
```

本地目录是 `project-portal/public/config/catalog.json`。其中 `allowHttpLocalhost=true` 只允许 loopback HTTP，不能放宽到局域网或公网 HTTP。

## 3. Catalog 契约

生产环境复制 `project-portal/config/catalog.example.json`，保存在部署系统的 ConfigMap/volume 中，不要把环境域名烘焙进 bundle。

```json
{
  "schemaVersion": 1,
  "updatedAt": "2026-07-22T00:00:00Z",
  "projects": [
    {
      "id": "my-project",
      "name": "项目名称",
      "summary": "一句公开说明",
      "category": "AI 平台",
      "capabilities": ["能力 A", "能力 B"],
      "tags": ["Java", "RAG"],
      "icon": "ai",
      "status": "available",
      "launchUrl": "https://app.example.com/login?returnTo=%2F",
      "healthUrl": "https://app.example.com/healthz",
      "openMode": "new-tab",
      "order": 40
    }
  ]
}
```

字段约束：

- `schemaVersion` 当前只能为 `1`；`projects` 最多 100 项，`id` 必须唯一。
- `status` 可取 `available`、`maintenance`、`coming-soon`。只有 `available` 生成链接。
- `openMode` 可取 `same-tab` 或 `new-tab`；新标签链接自动带 `noopener noreferrer`。
- 生产 `launchUrl` 必须是绝对 HTTPS URL，禁止 userinfo。未知 schema 或根结构错误会显示可重试错误；单个坏项目会被安全忽略并显示告警。
- `healthUrl` 必须与 `launchUrl` 同源，指向不需凭据的专用健康端点；端点需允许门户跨域 GET，只有 2xx 才会显示“可用”。
- Catalog 是匿名公开数据，只能放名称、说明和浏览器可达的目标登录地址；正式入口不预选 tenant/clientId。严禁 secret、token、内网 service DNS、管理地址、个人数据。

## 4. 目标项目租户选择契约

共同规则：

- 门户正式入口只进入目标 `/login`，不携带 `auto`、固定 tenant 或固定 clientId。
- `returnTo`/`redirect` 只允许目标 SPA 内的单斜杠绝对路径。外部 URL、`//`、反斜杠、控制字符会被拒绝或回退首页。
- 目标项目只接受自身已配置的 issuer、client 和 callback；门户不能覆盖这些安全配置，更不能直接拼 Casdoor authorize URL。
- tenant 必须在目标项目中输入或选择，再由目标项目映射到 allowlisted public client。兼容性的 `source=portal&auto=1` 可以保留，但不能作为新 catalog 的正式入口。

现有四个入口：

| 项目 | 正式 Launch URL 形式 | 租户校验 | 安全回退 |
| --- | --- | --- | --- |
| LangChain4j | `/login?redirect=%2F` | 用户输入 organization，并精确命中部署变量 `VITE_CASDOOR_TENANTS`；页面与 auth store 双层校验 | 未开放租户不发起 OIDC；空 allowlist 关闭提交；非 OIDC 模式保留原登录方式 |
| Recsys | `/login?returnTo=%2Foverview` | 当前只接受配置的 `recsys`，且 clientId 必须匹配该 org | 非法租户不发起 OIDC；legacy 保留原页 |
| Drools | `/ui/login?returnTo=%2Fhome` | 用户输入的 tenant 必须精确命中后端 `/activity-marketing/auth-config` 的 `webClients` | 未知租户与配置错误都不调用 `beginLogin` |
| Risk | `/login?returnTo=%2Fdashboard` | 当前只接受配置的 `risk-platform`，并校验派生 clientId 与 organization 一致 | 未知租户或配置漂移均不发起 OIDC；dev 模式保留本地身份入口 |

新增项目时应实现同等的目标-owned 租户页，并至少覆盖：未知租户不触发 OIDC、allowlist 映射正确、重复提交被禁止、回跳地址防开放重定向、配置关闭时不触发。若为兼容旧链接保留 auto adapter，还要继续覆盖合法参数只发起一次和非法/重复参数不触发。

LangChain4j 的匿名登录页不能直接读取 Casdoor 组织列表（组织 API 需要已登录身份），因此把可公开登录的 organization 作为构建期 allowlist。新增租户必须先在 Casdoor 完成组织、用户和 shared app 开通，再加入 `VITE_CASDOOR_TENANTS`/`SHOWCASE_CASDOOR_TENANTS` 并重建目标前端；不得为了临时放行恢复为“任意格式合法字符串都跳转”。

## 5. 生产域名与 Casdoor 配置

门户、目标前端、网关和 Casdoor 可以使用不同域名。点击入口是浏览器整页跨站导航，不依赖 CORS；但门户会跨域 GET 目标的 `healthUrl`，因此该匿名健康端点必须显式允许门户 origin（无敏感返回体时可用 `Access-Control-Allow-Origin: *`）。目标项目自身的业务 API 与 Casdoor CORS/redirect URI 仍按原部署配置。

每个目标项目必须在 Casdoor application 中精确登记它自己的 callback、silent renew 和 post-logout URI。例如生产域名改变时，应同时完成：

1. 在 Casdoor 更新目标项目 redirect URI/CORS origin；不要登记门户为业务 callback。
2. 更新目标项目的 issuer/client/base-path 配置并发布目标前端。
3. 只替换 portal 的运行时 `catalog.json` launch URL，不重新构建 portal 镜像。
4. 先以 `maintenance` 发布，完成 portal → target → Casdoor → callback 验证后再改为 `available`。

若目标入口位于子路径，URL 必须包含真实 base，例如 Drools 是 `/ui/login`。反向代理必须对 SPA deep link 提供 history fallback。

## 6. 容器发布与无重建换域名

```bash
docker build -t auth-platform/project-portal ./project-portal
docker run --rm -p 8203:80 \
  -v "$PWD/project-portal/config/catalog.prod.json:/usr/share/nginx/html/config/catalog.json:ro" \
  auth-platform/project-portal

curl -fsS http://localhost:8203/healthz
curl -fsS http://localhost:8203/config/catalog.json
```

nginx 对带 hash 的 assets 使用长期 immutable 缓存，对 catalog 使用 `no-store`，并设置 CSP、`Referrer-Policy: no-referrer`、防 framing 和 MIME sniffing 安全头。更新 ConfigMap/volume 后，新页面加载会读取新目录。

本地统一端口约定为 Recsys Console `9095`、Drools Gateway `8095`；页面“访问地址”直接来自 catalog 的 `launchUrl.host`，调整 Compose 端口时必须同步运行时 catalog。

## 7. 发布、回滚与排障

发布顺序：先发布目标 adapter（目录仍为 `maintenance`），再发布空/维护态门户，最后逐项目验证并开放。回滚单项目只需将它改为 `maintenance` 或 `enabled=false`；回滚门户则切回旧静态镜像。此功能没有数据库、SpiceDB schema 或后端 API 迁移。

常见问题：

- 点击后未显示租户选择：检查 catalog 是否仍含旧 `auto` 参数，以及目标前端是否已发布本版本。
- 输入租户后不能跳转：检查目标认证模式、租户 allowlist，以及 tenant/client/网关 organization 是否一致。
- Casdoor 报 redirect URI：修改的是目标项目 callback 白名单，不是门户地址。
- 登录后 401/403：门户只负责导航；检查目标网关 audience、owner/tenant、scope 与业务授权。
- 门户提示目录无效：用浏览器请求 `/config/catalog.json`，检查 schema、HTTPS、重复 id 和必填字段。
- 修改域名未生效：确认 volume 实际覆盖容器内 `/usr/share/nginx/html/config/catalog.json`，并检查代理没有覆盖 `Cache-Control: no-store`。

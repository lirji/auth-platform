# 新项目接入指南：其他项目接入 auth-platform 要改什么

> 回答一个问题：**「我名下一个新项目（或存量项目）要接统一权限平台，接入项目侧和平台侧各要改哪些东西？」**
> 平台有哪些能力见 [《平台能力总览》](./平台能力总览.md)；SSO 手把手细节见 [《统一登录平台接入手册》](./统一登录平台接入手册.md)；
> 项目要把**已有企业 IdP（上游 OIDC/SAML）联邦**汇入统一登录，见 [《企业SSO联邦接入指南》](./企业SSO联邦接入指南.md)。
> 参考实现：全链路样板 **langchain4j-platform**（knowledge-service ReBAC + edge Casdoor 换发 + 前端 OIDC）；
> **作用域继承型** **recsys**（资源不进 SpiceDB、反查归属再判 + 独立 SpiceDB 实例）和
> **对象归属型** **risk-platform**（共享 Casdoor + 本地 JWT 门禁 + 案件 assignee ReBAC，见 §6）。

---

## 0. 接入全景：三层能力，可分层采用

| 层 | 解决什么 | 平台组件 | 可否单独接 |
|---|---|---|---|
| **① 身份 / SSO** | 你是谁：统一登录、稳定 userId（`sub`）、租户（`owner`） | Casdoor（:8000） | ✅ 可只接这层 |
| **② 粗粒度 scope/角色** | 能不能进这个模块：token 里的 `groups`/`permissions` 直接判 | Casdoor（role→permission seed） | 依赖 ① |
| **③ 细粒度 ReBAC** | 能不能动**这一个**资源：关系元组判权 | SpiceDB + server(:8200) + sdk | 技术上可独立，但主体 id 应对齐 ①的 `sub` |

推荐顺序：**先 ①（身份对齐），再 ③（判权）**——③ 的关系元组主体必须写 Casdoor `sub`，身份没对齐判权会"空转"。② 按需。

## 1. 责任分工总表（一眼看清谁改什么）

### 平台侧（auth-platform / 平台管理员）要做的

| # | 事项 | 载体 | 必须/可选 |
|---|---|---|---|
| P1 | Casdoor 开租户：org（+ 用户），**每项目 = 一租户 = 一 org** | `deploy/casdoor-tenant-provision.sh`（方案C） | 必须 |
| P2 | Casdoor 应用：多租户项目走 Shared Application（派生 client_id `<base>-org-<tenant>`）；单租户项目独立建 app | 同上 / Casdoor 后台 | 必须（二选一） |
| P3 | 给项目一个**独立 SpiceDB 实例**（§B 决策：项目间不共享实例） | `deploy/docker-compose.yml` 翻版/新实例 | 必须 |
| P4 | 为项目**建模 `.zed` schema** 并灌入其 SpiceDB（新 definition 或复用 knowledge/his 心法） | `schemas/*.zed` + `POST /v1/schema/write` | 必须 |
| P5 | 部署该项目的判权面：`server`（必要时 `admin`+`auth-console`），`SPICEDB_HTTP/KEY` 指向 P3 实例 | server/admin 部署配置 | 必须 |
| P6 | server 写端加固：`authz.server.security.enabled=true` + 给消费方发 service token | env `AUTHZ_SERVER_SECURITY_ENABLED`/`AUTHZ_SERVER_TOKEN` | 生产必须 |
| P7 | 身份→授权同步：admin 配 `authz.casdoor.*`（organization=新 org、reconcile、department-sync 按需） | admin 配置 | 用组/部门模型才需要 |
| P8 | 粗粒度 scope 建模：role→permission seed（**先分配角色再 seed**） | `deploy/casdoor-seed.sh` | 用 ② 才需要 |
| P9 | seed/夹具 + 冒烟脚本（照 `dept-authz-fixture.sh`/`server-smoke.sh` 的样式给项目写一份） | `deploy/*.sh` | 强烈建议 |

### 接入项目侧要做的

| # | 事项 | 关键点 | 必须/可选 |
|---|---|---|---|
| C1 | 前端换统一登录：授权码+PKCE 跳 Casdoor | 手册 §4；redirect_uri 精确登记 | 接 ① 必须 |
| C2 | 后端换 OIDC resource-server：JWKS 验签 + iss/aud 校验 | 手册 §5；**`max-http-request-header-size: 64KB`** | 接 ① 必须 |
| C3 | 身份主键切到 `sub`（UUID）、租户只信 token `owner` | 存量数据要做 username→sub crosswalk | 接 ① 必须 |
| C4 | 引 SDK 依赖 + 配置 `authz.client.*` | 未发远程仓库：先本地 install；注意自定义 repo 路径 | 接 ③ 必须 |
| C5 | 实现 `SubjectResolver` bean（返回 `SubjectRef.user(<Casdoor sub>)`） | 无默认 bean，不提供则 `@CheckAccess` 不生效 | 接 ③ 必须 |
| C6 | 单点强制：业务方法标 `@CheckAccess`（需自引 `spring-boot-starter-aop`） | 判不过抛 `AccessDeniedException`→映射 403 | 推荐 |
| C7 | 列表过滤：检索/列表结果用 `checkBulk` 过滤（或 `lookupResources` 反查） | checkBulk 入参需去重；结果按入参顺序 | 有列表场景必须 |
| C8 | **资源生命周期双写**：创建资源时写归属元组，删除时删元组 | 见 §3.4；这是最容易漏的 | 接 ③ 必须 |
| C9 | 对象 id 加租户前缀 `<tenantId>_<id>` | 与平台约定一致，防跨租户串权 | 必须 |
| C10 | 写后读一致性：写返回的 ZedToken 回传 `at_least_as_fresh`（或 full） | 撤权立即生效场景必须 | 场景性 |
| C11 | 灰度开关：`disabled/shadow/enforce` 三态 + fail-closed | 引入即安全：默认 disabled，shadow 只观测 | 强烈建议 |

## 2. 平台侧改动详解

### 2.1 Casdoor：开租户 + 应用（P1/P2）

```bash
# 方案C 一条龙：幂等确保 shared app（rag-shared）+ 建 org + 建用户 + 验证 password grant
TENANT=myproj USER=alice PASSWORD='Alice@12345' bash deploy/casdoor-tenant-provision.sh
# 可选 WIRE_SPICEDB=1 顺带写 SpiceDB 成员组元组
```

- 登录/换 token 用**派生 client_id `<base>-org-<tenant>`**（共用 shared secret）；token `aud=[<base>-org-<tenant>]`、`owner=<tenant>`。
- 消费方网关/后端的 `aud` allowlist 一次性登记 base，按 `<base>-org-*` 家族 +（owner,aud）绑定校验——**之后新增租户零改动**。
- 单租户项目也可独立建 app（Casdoor 后台），按 `aud` 隔离与吊销；见手册 §3。
- **控制台菜单裁剪**：脚本会让新租户 org 继承 `built-in` 的 `navItems`。若已跑过 `casdoor-hide-business.sh`（隐藏「商业」等菜单），
  新开的每个租户 org 自动保持同一白名单，不会又冒出被隐藏的模块；built-in 未裁剪则不动。

### 2.2 SpiceDB：独立实例 + 项目建模（P3/P4）

- **每项目独立 SpiceDB 实例**（§B 决策，Casdoor 共享）。dev 直接翻版 `deploy/docker-compose.yml` 换端口/卷名；共享实例会导致 `schema/write` 整体替换误删他项目 definition。
- 建模三选一：
  1. **复用 knowledge 模型**（项目也是"空间/文档"形态）：直接灌 `knowledge.zed`，用部门层级或 space/folder 继承；
  2. **照 his.zed 样板写新 definition**（数据权限形态：科室/患者/病历 → 换成你的领域对象）；
  3. **全新建模**：心法见 `doc/getting-started-authz.md`（relation=手写的边，permission=并集+箭头推导；RBAC=group 绑 relation）。
- 灌入：`cat your.zed | jq -Rs '{schema:.}' | curl -X POST $SPICEDB_HTTP/v1/schema/write ...`（参考 `his-smoke.sh` 的合并写法）。

### 2.3 判权面部署 + 加固（P5/P6/P7）

```yaml
# 该项目的 server 实例（:8200 或新端口）
SPICEDB_HTTP: http://<该项目 spicedb>:8543
SPICEDB_KEY:  <该实例 preshared key>
AUTHZ_SERVER_SECURITY_ENABLED: "true"     # 生产必须；enabled 而 token 空会启动失败（fail-fast）
AUTHZ_SERVER_TOKEN: <发给消费方的 service token>
```

- 需要授权管理界面/组同步时再部署 admin（+auth-console），admin 配 `authz.security.*`（Casdoor JWKS/issuer/clientId）与 `authz.casdoor.*`：`enabled=true`、`organization=<该项目 org>`、`reconcile-enabled=true`（组同步定时对账）、用部门模型再开 `department-sync-enabled=true`。一般**一项目一 org 一 admin 实例**；单实例要同步多个 org 时配 `authz.casdoor.organizations` 列表（组同步与部门同步都逐 org 拉取合并，组/部门 id 天然带 `<org>_` 前缀无碰撞；`delete-threshold` 仍是跨 org 累计的全局熔断，大迁移注意调大）。
- 部门管理员约定：Casdoor role `<group>-admin` 的 users = 部门 `<group>` 的管理员（`DepartmentSyncService` 读取）。

### 2.4 scope seed 与夹具（P8/P9）

- 粗粒度 scope：`CASDOOR_CLIENT_ID=... CASDOOR_CLIENT_SECRET=... CASDOOR_ORG=<org> bash deploy/casdoor-seed.sh`。**必须先把角色分配给用户再跑**（Casdoor permission 展开在写入时定死，反了要重跑）。
- 给项目照 `dept-authz-fixture.sh` 写一份 seed+自校验脚本（默认 dry-run、`APPLY=1` 写入、写后强一致断言、失败非零退出）——这是本仓库事实上的验收方式。

## 3. 接入项目侧改动详解

### 3.1 身份（C1-C3）——见手册，此处只列改动面

- **前端**：登录页/token 管理换 `oidc-client-ts`（授权码+PKCE），可直接抄 `auth-console/src/auth/`。多租户项目 client_id 用 `<base>-org-<tenant>`（登录页输入租户）。
- **后端**：加 `spring-boot-starter-oauth2-resource-server`，抄 `auth-platform-admin/.../SecurityConfig.java`（JwtDecoder：JWKS+iss+aud；groups 归一化）。**务必配 `server.max-http-request-header-size: 64KB`**（Casdoor token ~9KB，否则合法 token 400）。
- **数据**：用户主键切 `sub`；`tenantId` 只取 token `owner`，拒绝 header/query 覆盖；存量 username 数据做一次性 crosswalk。

### 3.2 判权依赖与配置（C4/C5）

```xml
<dependency>
  <groupId>com.lrj.authz</groupId>
  <artifactId>auth-platform-sdk</artifactId>
  <version>0.1.0-SNAPSHOT</version>
</dependency>
<!-- 用 @CheckAccess 才需要（SDK 内是 optional，不传递）： -->
<dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-aop</artifactId>
</dependency>
```

> ⚠️ SDK 未发远程仓库：先在本仓库 `./mvnw -pl auth-platform-protocol,auth-platform-sdk install -DskipTests`。
> **若消费方项目用自定义本地仓库**（如 langchain4j 用 `/Users/liruijun/personal/repository`），install 必须加
> `-Dmaven.repo.local=<那个仓库>`，否则消费方编译到的是旧 SDK（症状：`找不到符号`）。

```yaml
authz:
  client:
    server-url: http://<该项目 server>:8200
    token: ${AUTHZ_CLIENT_TOKEN:}   # = 平台侧 P6 发的 service token
    # enabled 默认就是 true；connect-timeout 2s / read-timeout 5s 可调
```

```java
// SubjectResolver：无默认 bean，必须提供；返回值必须是 Casdoor sub（与元组主体对齐）
@Bean
SubjectResolver subjectResolver() {
    return () -> SubjectRef.user(currentJwt().getSubject());
}
```

### 3.3 判权四姿势（C6/C7）

```java
// ① 单点强制（推荐默认姿势）：参数名匹配（非 SpEL），判不过抛 AccessDeniedException
@CheckAccess(permission = "view", resourceType = "document", resourceIdParam = "docId")
public Document getDocument(String docId) { ... }
// 撤权需立即生效的敏感操作：@CheckAccess(..., fullyConsistent = true)

// ② 手动单查（分支逻辑里用）
boolean ok = engine.check(subject, "share", ResourceRef.of("document", id), Consistency.minimizeLatency());

// ③ 列表过滤（检索结果后过滤；入参需先去重）
Map<ResourceRef, Boolean> allowed = engine.checkBulk(subject, "view", refs, consistency);

// ④ 反查（"我能看哪些" 直接问引擎，替代全表扫）
List<String> ids = engine.lookupResources(subject, "view", "document", consistency);
```

异常映射：`AccessDeniedException` → HTTP 403（`@ExceptionHandler` 或 `ResponseStatusException`）。
协议异常（server/SpiceDB 响应异常时 SDK 抛 `IllegalStateException`）**不要捕获吞掉折成 deny/allow**——让它冒泡，enforce 自然 fail-closed。

### 3.4 资源生命周期双写（C8/C9，最容易漏）

判权只认 SpiceDB 里的边——**业务创建/删除资源时必须同步写/删元组**，否则新资源无人可见（或删除后残留幽灵授权）：

```java
// 创建文档时（参考 langchain4j knowledge-service 的 onDocumentCreated）
engine.writeRelationships(List.of(
    RelationshipUpdate.touch(ResourceRef.of("document", tenantId + "_" + docId), "owner",
                             SubjectRef.user(sub)),
    RelationshipUpdate.touch(ResourceRef.of("document", tenantId + "_" + docId), "home_dept",
                             SubjectRef.of("department", tenantId + "_" + dept))));
// 删除资源时
engine.deleteRelationships(RelationshipFilter.ofResource(ResourceRef.of("document", tenantId + "_" + docId)));
```

- 对象 id 一律 `<tenantId>_<id>`；部门/组 id `<org>_<group>`（与 `CasdoorGroupIds` 同构）。
- enforce 下建资源缺归属（如无部门）→ 建议直接 403 拒建，不写孤儿资源（langchain4j 的守卫姿势）。
- 单篇分享/撤销 = 写/删 `document#viewer@user:<sub>` 一条边（可走 admin `/admin/grants` 或自己 `writeRelationships`）。

### 3.5 写后读与灰度（C10/C11）

- 写返回 `ZedTokenView.token`，随后的判权带 `Consistency.atLeastAsFresh(token)`——"授权后立刻可见/撤权立即失效"必须这样，否则可能被量化快照漏读。
- 灰度三态（照 langchain4j `app.rag.authz.mode` 的成熟姿势）：`disabled`（与接入前逐字一致）→ `shadow`（只记指标不拦截，观测 deny 率）→ `enforce`（真过滤，fail-closed）。**默认 disabled，引入即安全**；shadow 期把 deny 样本核对干净再切 enforce。

## 4. 验证清单（分层，每层过了再上一层）

| 层 | 验什么 | 怎么验 |
|---|---|---|
| 0 | Casdoor 可达、JWKS 正常 | `bash deploy/sso-smoke.sh`（Layer 0） |
| 1 | 换到真 token、claim 契约对（sub/owner/groups） | `sso-smoke.sh` Layer 1（带 CLIENT_ID/SECRET） |
| 2 | 你的后端验签矩阵（无 token 401 / 篡改 401 / 合法 200 或 403） | `AUTHZ_ADMIN=http://你的服务 ADMIN_SMOKE_PATH=/你的受保护端点 bash deploy/sso-smoke.sh` |
| 3 | SpiceDB 模型正确（seed + 强一致断言） | 项目版 fixture 脚本（照 `dept-authz-fixture.sh`） |
| 4 | 经 server 的判权链路 + ZedToken 写后读 | 照 `server-smoke.sh` |
| 5 | 业务端到端：owner 可见 / 他人不可见 / 授权即见 / 撤权即失 | 集成测试（参考 langchain4j `KnowledgeAuthzIntegrationTest`） |

## 5. 常见坑速查

| 坑 | 对策 |
|---|---|
| 判权永远 deny，元组看着都对 | `SubjectResolver` 返回的不是 Casdoor `sub`（写的 `user:<sub>`，判的却是 username）——身份先对齐 |
| 消费方编译报"找不到符号"但 SDK 明明改了 | 双 maven 仓库：install 时没加 `-Dmaven.repo.local=<消费方的仓库>` |
| `@CheckAccess` 完全不生效 | 缺 `spring-boot-starter-aop` 或没有 `SubjectResolver` bean（切面装配是三条件：aspectj 在类路径 + 有 resolver + 无同名 bean） |
| `resourceIdParam` 报 `IllegalStateException` | 参数名对不上（切面按**编译期参数名**匹配，非 SpEL；Boot parent 默认 `-parameters`，自定义编译配置需保留） |
| 合法 token 400（不是 401） | 没配 `server.max-http-request-header-size: 64KB`（Casdoor token ~9KB） |
| 新建的资源谁都看不见 | 漏了 §3.4 生命周期双写（创建时没写 owner/归属边） |
| 授权/撤权"不生效" | 写后读没带 ZedToken；调试时用 full 一致性复核 |
| 组同步后嵌套组成员被删 | 已由 `readRelationships` direct-tuple 差量修复；自写同步逻辑勿用 `lookupSubjects` 比对 |
| 大批删除元组 | `authz.casdoor.delete-threshold`（默认 1000）会熔断整轮；确认是预期迁移再临时调大 |
| `permissions` claim 为空 | seed 顺序：先给用户分配角色，再跑 `casdoor-seed.sh` |
| server 写端裸奔 | 生产必须 `authz.server.security.enabled=true`（默认关）+ SDK 侧配对应 `authz.client.token` |

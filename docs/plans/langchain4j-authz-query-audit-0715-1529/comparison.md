# Candidate Comparison

> 视角：risk-reviewer + plan-judge。5 分代表更有利；“改动风险/复杂度/测试难度/回滚成本”均按风险更低、复杂度更低、测试更容易、回滚更便宜得高分。

## 2. 关键失败场景对比

| 风险面 | A | B | C |
|---|---|---|---|
| malformed SDK response | 直接修复 | 仍需合并 A 修复 | 仍需修复 lookup/check 协议 |
| AP/SpiceDB 故障 | 延续已验证 fail-open/closed | 可统一指标与错误，但新分支更多 | lookup 失败将阻断整次召回 |
| shadow share | 沿用 endpoint 不暴露 | 可观测，但可能改变 API/写语义 | 不是主要解法 |
| D3 tuple 缺失 | 发布门禁阻断 | 同样需要门禁 | lookup 返回不全，仍误拒 |
| semantic cache 绕过 | 先硬禁冲突组合 | 仍需硬禁或加门面版本 | 可长期解决，但系统复杂 |
| 大租户性能 | bounded bulk，风险可测 | 同 A | lookup/大 IN 集风险最高 |
| 撤权即时生效 | full check 正确；缓存关闭 | 同 A | 依赖 epoch/重校验正确实现 |

## 3. 风险审查结论

- 兼容性：A 不改合法协议与外部 API；B 可能改变 shadow 404；C 改内部检索契约和排序。
- 事务/幂等：A 仅复用 TOUCH 与只读验证；B 要定义 shadow 写失败；C 引入 tuple+epoch 双写。
- 并发：A 当前 bulk 分批稳定，任何批失败整次安全降级；B 身份快照更好；C 要处理授权版本竞态。
- 性能：full consistency 是 A/B 的主要成本；C 的 lookup 高基数更不可控。
- 安全：A 足以关闭已知越权旁路条件；B 改善设计一致性；C 理论边界最好但实现错误面最大。
- 数据迁移：A 仅关系 backfill/preflight；B 无数据迁移；C 需缓存/epoch 迁移。
- 灰度/回滚：A 最简单，切 shadow 即回滚；B 需同时回滚 bean 条件；C 要双路径与数据兼容。

## 4. 选择

选择 **A 为最终实施方案**，吸收 B 的两个低风险优点作为可选加固：方法入口身份快照、AOP dependency-error 测试/指标。暂不吸收 B 的 shadow endpoint 变化，也不吸收 C 的 lookup/ACL 下推。

选择弱点是长期依赖发布门禁且 semantic cache 只能先关闭；这是有意接受的范围约束，不宣称问题已永久消失。

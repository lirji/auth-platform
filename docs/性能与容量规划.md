# 性能与容量规划：高用户体量下平台能不能接住？

> 回答一个问题：**判权平台在高并发下的承载能力、瓶颈在哪、按什么顺序优化。**
> 结论先行：**当前架构在 ~1k QPS 判权量级内无压力；1k~10k 靠水平扩展与一致性档位纪律；再往上
> 的第一性瓶颈是「双 HTTP/JSON 跳」和「SpiceDB datastore」，前者换 gRPC 适配器（端口已预留），
> 后者上 SpiceDB dispatch 集群 / 换分布式 datastore。** 平台无状态组件皆可横向扩，没有单点设计死角。
> （2026-07-16 结论，基于当前代码；已随本轮落地的优化见 §5。）

---

## 1. 判权链路成本解剖（一次 check 都花在哪）

```
业务代码 → sdk RemoteAuthzEngine ──HTTP/JSON──▶ server(:8200, Tomcat 阻塞式)
              └── ①序列化+连接             ②线程占用
         → SpiceDbAuthzEngine ──HTTP/JSON──▶ SpiceDB HTTP 网关(grpc-gateway 转译)
              └── ③再一次序列化+连接        ④JSON→protobuf 转译
         → SpiceDB 核心(缓存/子问题分发) → Postgres(datastore)
              └── ⑤缓存命中≈亚毫秒          ⑥缓存不命中→查库,毫秒级
```

量级参考（本地实测/官方口径，生产以压测为准）：

| 环节 | 成本 | 说明 |
|---|---|---|
| SpiceDB check（缓存命中） | ~1-5ms | `MINIMIZE_LATENCY` 走量化快照缓存 |
| SpiceDB check（`FULLY_CONSISTENT`） | ~5-20ms+ | **绕过全部缓存直打 datastore**，高并发下最贵 |
| HTTP 网关转译 + JSON | ~1-2ms/跳 | 共两跳（sdk→server、server→SpiceDB） |
| Tomcat 线程 | 200 worker（默认） | 阻塞式：吞吐 ≈ 线程数 ÷ 单请求耗时 |

粗算单实例 server：端到端 5-10ms/check、200 线程 → **理论 2 万-4 万 QPS 天花板**，但实际瓶颈
在此之前出现（见 §2）。`checkBulk` 把 N 篇文档折成一次请求，是列表场景的第一优化（已是 SDK 标准姿势）。

## 2. 瓶颈清单（按出现顺序）与对策

| # | 瓶颈 | 症状 | 对策 | 状态 |
|---|---|---|---|---|
| B1 | ~~HTTP 客户端每主机 5 连接~~（`HttpURLConnection` JVM 默认） | 并发 >5 时建连排队、延迟锯齿 | 换 JDK HttpClient（连接池化 keep-alive），sdk+core 已换 | ✅ 本轮已修 |
| B2 | ~~core 无超时~~ | SpiceDB 故障时请求线程无限挂起 → 线程耗尽雪崩 | connect 2s / read 15s（可配 `authz.spicedb.connect-timeout/read-timeout`） | ✅ 本轮已修 |
| B3 | `FULLY_CONSISTENT` 滥用 | SpiceDB CPU/PG 压力陡增、p99 恶化 | **判权纪律**：常规判权 `minimize_latency`；写后读用 ZedToken `at_least_as_fresh`（server 现有水位缓存兜底无 token 的请求）；full 只留给 admin 管理台（低频） | ✅ 水位已落地 |
| B4 | 列表场景 N+1 check | QPS 放大 N 倍 | 禁止逐篇 check 循环：`checkBulk`（单请求）或 `lookupResources` 预过滤下推到 ES/DB | 姿势已文档化 |
| B5 | SpiceDB 量化窗口 | 缓存命中率不足 | 调 `--datastore-revision-quantization-interval`（默认 5s）：调大→命中率↑、撤权可见延迟↑。撤权敏感操作本来就该带 token/full，所以窗口可以放心调大 | 部署参数，按压测调 |
| B6 | server 单实例线程/CPU | QPS 到几千后饱和 | server **无状态**：水平扩 N 实例 + LB 即可；单实例可调 `server.tomcat.threads.max`。注意：水位缓存是单实例内存态，多实例下跨实例写后读仍须调用方自带 ZedToken（SDK 本来就这么用） | 架构已支持 |
| B7 | SpiceDB 单实例 | 缓存装不下、CPU 饱和 | SpiceDB **dispatch 集群**：多副本 + 一致性哈希分发子问题，缓存分片复用（官方水平扩展方案）；`--dispatch-upstream-addr` | 部署形态，未到量级 |
| B8 | Postgres datastore | 连接数打满、GC 落后 | pgbouncer 连接池；`--datastore-conn-pool-*` 调优；再往上换 CockroachDB/Spanner（SpiceDB 原生支持）才是真·水平扩展 | 未到量级 |
| B9 | 双 HTTP/JSON 跳的固有开销 | p99 里稳定的 2-4ms 税 | **gRPC 适配器**：`AuthzEngine` 端口保留正为此——新增一个 gRPC 实现类，上层零改。前提：消费方依赖树能容纳 grpc/protobuf（langchain4j 的钻石冲突正是当年选 HTTP 的原因，换前先验依赖） | 规划项（刻意未做） |
| B10 | 应用层判权结果缓存 | —— | **默认不要做**：撤权时效会被 TTL 决定（等于自己造一致性坑）。SpiceDB 自身缓存已经覆盖热点；真要做，参考 langchain4j `SemanticCache` 教训（F1：按租户分桶+默认关） | 反模式警示 |

### 登录面（Casdoor）单独说

判权面高频、登录面低频（token 有效期内后端只做**本地 JWKS 验签**，零网络调用）。Casdoor 只在
登录/续期时被打到，容量压力比判权面低 1-2 个数量级；扩容 = Casdoor 多实例 + 共享 PG。
`NimbusJwtDecoder` 自带 JWKS 缓存，Casdoor 短暂抖动不影响存量 token 验签。

## 3. 体量分档路线图

| 判权量级 | 部署形态 | 需要做什么 |
|---|---|---|
| **~1k QPS**（当前目标） | server×1 + SpiceDB×1 + PG×1（现状 compose） | 无需动作。B1/B2 修复后单实例即可承接；保持一致性档位纪律（B3） |
| **1k~10k QPS** | server×2-4 + LB；SpiceDB×2-3 | server 直接复制实例；SpiceDB 开 dispatch；PG 加 pgbouncer；量化窗口按撤权 SLA 调大；开始压测建基线（见 §4） |
| **10k~100k QPS** | gRPC 适配器 + SpiceDB dispatch 集群 + CockroachDB | B9 落地（新增 gRPC `AuthzEngine` 实现）；datastore 换分布式；**§B 每项目独立 SpiceDB 本身就是天然分片**——单项目内再谈集群 |
| **热点读极端场景** | 消费方 `lookupResources` 物化 | 把"我能看哪些"预查出来下推到检索引擎过滤（langchain4j 已是此姿势），判权面只兜底 spot-check |

## 4. 压测与可观测

- **指标源**：SpiceDB Prometheus（宿主 `:9099`）——重点看 `spicedb_datastore_query_latency`、cache hit 率、dispatch 数；server/admin actuator health；PG 连接数。
- **压测姿势**（建基线，改动前后对比）：
  ```bash
  # 单条 check 压测样例(vegeta/k6 均可;先 seed 再压)
  echo 'POST http://localhost:8200/v1/check
  Content-Type: application/json
  @check.json' | vegeta attack -rate=500 -duration=30s | vegeta report
  # check.json: {"subject":{"type":"user","id":"u_bob"},"permission":"view",
  #              "resource":{"type":"document","id":"d_42"},"consistency":{"mode":"minimize_latency"}}
  ```
  断言建议：`minimize_latency` p95 < 15ms、`full` p95 < 50ms、错误率 0；对比开/关水位、不同量化窗口。
- **容量红线**：Tomcat busy threads 持续 > 70%、SpiceDB cache hit < 80%、PG 连接 > 80% 池上限——任一触线进入下一档扩容。

## 5. 本轮已落地的优化/硬化（2026-07-16）

| 项 | 内容 |
|---|---|
| HTTP 连接池 | sdk `RemoteAuthzEngine` + core `SpiceDbAuthzEngine` 从 `SimpleClientHttpRequestFactory`（每主机 5 连接）换 JDK HttpClient（池化 keep-alive） |
| 超时全覆盖 | core 原先**无任何超时**；现 connect 2s / read 15s（server）/ 30s（admin），可配 |
| ZedToken 水位缓存 | server 记录最近写水位；`at_least_as_fresh` 无 token 的请求自动代入（原先被 SpiceDB 拒绝），无水位安全回退 full。开关 `authz.server.zed-token-watermark-enabled`（默认开；多实例边界见 B6） |
| 审计持久化 | admin 审计可落 Postgres（`authz.audit.persistence-enabled`，默认关=内存 500 条）；重启不丢，带 retention 裁剪 |
| 同步运维性 | 部门同步新增 HTTP 触发端点 + webhook 联动；Casdoor 同步支持多 org（`authz.casdoor.organizations`） |

> 刻意未做：gRPC facade（B9，等量级到了再上，且须先验消费方依赖树）；应用层判权缓存（B10，反模式）。

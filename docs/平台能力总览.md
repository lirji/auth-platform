# auth-platform 能力总览（Capability Reference）

> 面向对象：想知道「这个平台到底能干什么、对外暴露哪些接口、边界在哪」的接入方与维护者。
> 全部内容来自当前代码实测（2026-07-18，main @ e12c2d4 之后，含 recsys 模型 + 全项目补测 + ADM01 两段审计 + fail-closed 收紧），非规划态；规划态项在 [§10 已知边界](#10-已知边界与未落地项) 显式标注。
> 接入操作步骤见姊妹篇 **[《新项目接入指南》](./新项目接入指南.md)**；SSO 细节见 **[《统一登录平台接入手册》](./统一登录平台接入手册.md)**；企业 IdP 联邦（上游 SSO 汇入 Casdoor）见 **[《企业SSO联邦接入指南》](./企业SSO联邦接入指南.md)**。

## 1. 平台定位与能力矩阵

**内部统一权限平台（IAM）= Casdoor（身份/登录/SSO，OIDC）+ SpiceDB（Zanzibar/ReBAC 细粒度授权）。**
本平台是纯授权平台：不含业务代码、无应用层数据库，所有授权态都是 SpiceDB 里的关系元组。

| 能力域 | 提供什么 | 载体 |
|---|---|---|
| **身份 / SSO** | OIDC 登录、RS256 JWT（`sub`=全局用户ID、`owner`=租户、`groups`/`permissions`）| Casdoor（:8000，外部服务） |
| **判权（decision）** | check / checkBulk / lookupResources / lookupSubjects / expand | `auth-platform-server`（:8200，`/v1/*`） |
| **授权管理（administration）** | grant/revoke、关系元组读写、schema 查看、判权调试、审计 | `auth-platform-admin`（:8201，`/admin/*`）+ `auth-console` 前端 |
| **身份→授权同步** | Casdoor 组成员 / 部门树 → SpiceDB 元组（差量、幂等、熔断） | admin 的 `GroupSyncService` / `DepartmentSyncService` / `ReconcileJob` / webhook |
| **消费方接入** | Spring Boot Starter：`AuthzEngine` bean + `@CheckAccess` 声明式判权 | `auth-platform-sdk` |
| **多租户** | 对象 id `<tenantId>_<id>` 前缀；组/部门 id `<org>_<group>` 强制编码；Casdoor 每项目一 org；方案C Shared Application | 约定 + `CasdoorGroupIds` + `casdoor-tenant-provision.sh` |
| **授权模型** | `.zed` schema：知识库（部门层级隔离）+ HIS 数据权限样板 + recsys 广告主作用域（另库） | `auth-platform-core/src/main/resources/schemas/` |

架构（六边形，一个端口两个适配器）：

```
消费方业务代码 ──@CheckAccess/AuthzEngine──▶ sdk(RemoteAuthzEngine) ──HTTP──▶ server(:8200) ──┐
auth-console ──▶ admin(:8201) ──────────────────────────────────────────────┐               │
Casdoor(:8000) ──GroupSync/DeptSync──▶ admin ───────────────────────────────┴─ SpiceDbAuthzEngine ──HTTP/JSON──▶ SpiceDB(:8543)
```

依赖方向：`protocol ← core ← {server, admin}`，`protocol ← sdk`。全程 **grpc-free**（SpiceDB 走 HTTP/JSON）。

## 2. `AuthzEngine` 端口（9 个操作）

`auth-platform-protocol` 定义、两个实现（core `SpiceDbAuthzEngine` 直连 SpiceDB；sdk `RemoteAuthzEngine` 经 server）。换引擎/加 gRPC 适配器只需新增实现类。

| # | 操作 | 签名 | 语义 |
|---|---|---|---|
| 1 | check | `boolean check(SubjectRef, String permission, ResourceRef, Consistency)` | 单条判权 |
| 2 | checkBulk | `Map<ResourceRef,Boolean> checkBulk(SubjectRef, permission, List<ResourceRef>, Consistency)` | 批量判权，保持入参顺序（列表过滤用） |
| 3 | lookupResources | `List<String> lookupResources(SubjectRef, permission, resourceType, Consistency)` | 反查「我能看哪些资源」 |
| 4 | lookupSubjects | `List<SubjectRef> lookupSubjects(ResourceRef, permission, subjectType, Consistency)` | 反查「谁能看这个资源」 |
| 5 | writeRelationships | `ZedTokenView writeRelationships(List<RelationshipUpdate>)` | 写元组（CREATE/TOUCH/DELETE），返回一致性水位 |
| 6 | deleteRelationships | `ZedTokenView deleteRelationships(RelationshipFilter)` | 按过滤器批量删 |
| 7 | readSchema | `String readSchema()` | 读当前 `.zed` schema 文本 |
| 8 | expand | `String expand(ResourceRef, permission, Consistency)` | 展开判定树（调试「权限怎么推出来的」） |
| 9 | readRelationships | `List<Relationship> readRelationships(RelationshipFilter)` | 读原始 direct 元组（同步差量比对用；core 侧强制 full 一致性） |

核心 record：`ResourceRef(type,id)`、`SubjectRef(type,id,relation)`（relation 非空即 userset 主体如 `group:eng#member`）、`RelationshipUpdate(op,resource,relation,subject)`（`TOUCH`=幂等写）、`RelationshipFilter(resourceType,resourceId?,relation?)`（null=通配）、`Consistency(mode,zedToken)`、`ZedTokenView(token)`。

**严格响应校验（fail-closed，勿放松）**：sdk `requireAllowed`/`parseCheckBulk` 与 core `hasPermission`/checkBulk `pairs` 基数校验对称——响应缺字段/错位/per-item error **一律抛 `IllegalStateException`**，不折成 deny；上层 enforce 据此 fail-closed。**2026-07-18 再收紧**（对称覆盖全部读写路径）：core `lookupResources` 缺/空 `permissionship` 抛（不再当作有权限）、`writeRelationships`/`deleteRelationships` 缺 `writtenAt.token`/`deletedAt.token` 抛（`requireToken`——无水位=read-after-write 无法保证）、流式响应含顶层 `error` 抛（不返可信部分结果）；server `check-bulk` 引擎响应缺某资源直接抛（不再 `Boolean.TRUE.equals(null)` 静默降级成 `allowed=false`，否则把依赖故障伪装成 deny 且 SDK 严格校验无从察觉）。

## 3. server（:8200）判权 REST API

`AuthzController`，`/v1/*`，请求/响应 JSON 与 protocol record 同构。

| 方法 | 路径 | 请求体要点 | 响应 | 对应操作 |
|---|---|---|---|---|
| POST | `/v1/check` | `{subject, permission, resource, consistency}` | `{allowed}` | check |
| POST | `/v1/check-bulk` | `{subject, permission, resources[], consistency}` | `{results:[{resource,allowed}]}`（按入参顺序） | checkBulk |
| POST | `/v1/lookup-resources` | `{subject, permission, resourceType, consistency}` | `{resourceIds[]}` | lookupResources |
| POST | `/v1/lookup-subjects` | `{resource, permission, subjectType, consistency}` | `{subjects[]}` | lookupSubjects |
| POST | `/v1/relationships` | `{updates[]}` | `{token}` | writeRelationships |
| POST | `/v1/relationships/delete` | `{filter}` | `{token}` | deleteRelationships |
| POST | `/v1/relationships/read` | `{filter}` | `{relationships[]}` | readRelationships |
| GET | `/v1/schema` | — | `{schema}` | readSchema |
| POST | `/v1/expand` | `{resource, permission, consistency}` | SpiceDB 展开树原文 JSON | expand |

`consistency` 传法：`{"mode":"minimize_latency"|"at_least_as_fresh"|"full", "zedToken":"..."}`（省略即 minimize_latency；`fully_consistent` 也识别为 full）。

**安全**：`AuthzServerSecurityFilter`（非 spring-security）只守 `/v1/*`——`Authorization: Bearer <service token>` 常量时间比较；配置 `authz.server.security.enabled`（**默认 false**）+ `authz.server.security.token`（env `AUTHZ_SERVER_SECURITY_ENABLED` / `AUTHZ_SERVER_TOKEN`）；enabled 而 token 空 → 启动失败（fail-fast）。⚠️ 生产必须开启，否则 `/v1/relationships` 写端裸奔。

**SpiceDB 连接**：`authz.spicedb.endpoint`（env `SPICEDB_HTTP`，默认 `http://localhost:8543`）+ `authz.spicedb.token`（env `SPICEDB_KEY`，默认 `authz_dev_key`）+ `connect-timeout`/`read-timeout`（默认 2s/15s；admin 侧 read 30s）。底层 JDK HttpClient 连接池。

**ZedToken 水位缓存**：server 记录本实例最近写返回的 ZedToken；`at_least_as_fresh` 请求没带 token 时自动代入水位（原先会被 SpiceDB 拒绝），无水位安全回退 full。开关 `authz.server.zed-token-watermark-enabled`（默认开）。⚠️ 单实例内存态——多实例部署跨实例写后读仍应由调用方回传自己的 ZedToken。

## 4. admin（:8201）授权管理 REST API

`AdminController` + `CasdoorSyncController`。**读一律 `FULLY_CONSISTENT`**（授予→立即反查一致）。写操作走**两段审计（ADM01）**落审计存储。

| 方法 | 路径 | 功能 | 所需权限 |
|---|---|---|---|
| POST | `/admin/grants` | 授予一条关系（TOUCH 幂等）：`{resourceType,resourceId,relation,subjectType,subjectId,subjectRelation?}` | `authz-admin` |
| POST | `/admin/grants/revoke` | 撤销一条关系 | `authz-admin` |
| GET | `/admin/resources/{type}/{id}/subjects?permission=&subjectType=user` | 反查主体 | admin/viewer |
| GET | `/admin/subjects/{type}/{id}/resources?permission=&resourceType=` | 反查资源 | admin/viewer |
| GET | `/admin/schema` | 读 `.zed` schema | admin/viewer |
| POST | `/admin/check` | 判权调试：`{subjectType,subjectId,permission,resourceType,resourceId}` → `{allowed}` | admin/viewer |
| POST | `/admin/expand` | 展开判定树（用 resource+permission，忽略 subject 字段） | admin/viewer |
| GET | `/admin/relationships?resourceType=&resourceId=&relation=` | 列现存 direct 元组（resourceId/relation 可省） | admin/viewer |
| GET | `/admin/audit?limit=100` | 最近审计记录 | admin/viewer |
| POST | `/admin/casdoor/sync` | 手动触发**组同步**（`enabled=false` → 409） | `authz-admin` |
| POST | `/admin/casdoor/sync-departments` | 手动触发**部门树同步**（`department-sync-enabled=false` → 409） | `authz-admin` |
| POST | `/admin/casdoor/webhook` | Casdoor 回调触发组同步（部门同步已启用时一并触发）；header `X-Webhook-Secret` 常量时间校验 | 密钥（permitAll + secret） |

**安全**：OIDC resource-server 校验 Casdoor JWT（JWKS + issuer + 可选 aud），`groups` claim 归一化（`built-in/authz-admin`→`authz-admin`）为 authority。配置前缀 `authz.security.*`：`jwk-set-uri`（env `CASDOOR_JWKS`）、`issuer`（`CASDOOR_ISSUER`）、`client-id`（`CASDOOR_CLIENT_ID`，空=跳过 aud 校验）、`webhook-secret`（`AUTHZ_WEBHOOK_SECRET`，空=不校验）。`max-http-request-header-size: 64KB` 已配（Casdoor token ~9KB）。

**审计 `AuditStore`**（端口 + 两实现，按 `authz.audit.persistence-enabled` 二选一）：默认内存环形缓冲
500 条（重启即失）；开启持久化后落 Postgres `authz_audit` 表（独立库 `authz_admin`，启动幂等自建表、
DB 不可达则 fail-fast 拒绝启动，`retention-max-rows` 裁剪，默认 10 万行）。配置 `authz.audit.*`：
`persistence-enabled`（默认 false）、`jdbc-url`（env `AUTHZ_AUDIT_JDBC_URL`）、`username`/`password`、
`retention-max-rows`、`in-memory-capacity`。记录 `{at,actor,action,detail}`（actor 取 JWT `name`/`sub`），
仅 grant/revoke 落审计。dev 建库：新 compose 卷由 `deploy/postgres-init/` 自动建，存量卷手动
`docker exec authz-postgres createdb -U authz authz_admin`。

**两段审计（ADM01，勿改成一段）**：grant/revoke 对 SpiceDB 的数据面写与审计库是两个独立系统（无共享事务），
为杜绝"数据已变却无审计"，一次写拆两段——写前先落 `<action>.intent`，写成功落 `<action>.ok`，写失败落
`<action>.fail`。`intent` **故意不捕获**：审计库不可用时直接失败、绝不写 SpiceDB（fail-closed，审计是安全记录、
拒绝静默降级）；`ok`/`fail` 收尾用 best-effort（审计写失败只 ERROR 记账、不掩盖原始写异常）。因此一次 grant 在
`/admin/audit` 里表现为 `grant.intent`→`grant.ok` 两条（失败为 `grant.intent`→`grant.fail`）。

## 5. sdk（消费方 Starter）

坐标 `com.lrj.authz:auth-platform-sdk:0.1.0-SNAPSHOT`（Boot 3.3.5 / Java 21；**未发中央仓库**，需本地 `install`）。

- **自动配置**：`AutoConfiguration.imports` 注册；开关 `authz.client.enabled` **默认开**（matchIfMissing=true，与平台"开关默认关"约定不同，注意）。提供 `AuthzEngine` bean（=`RemoteAuthzEngine`，`@ConditionalOnMissingBean` 可覆盖为 Noop 等）。
- **配置项**（前缀 `authz.client`）：`server-url`（默认 `http://localhost:8200`）、`token`（默认空→不带 Authorization 头）、`connect-timeout`（2s）、`read-timeout`（5s）、`enabled`（true）。
- **`@CheckAccess(permission, resourceType, resourceIdParam, fullyConsistent=false)`**：AOP 切面从**方法参数名**（非 SpEL，依赖 `-parameters`，Boot parent 默认开）取资源 id → `engine.check()`；不通过抛 `AccessDeniedException`（建议映射 403）。生效需两个额外条件：消费方自引 `spring-boot-starter-aop`（SDK 里是 optional）+ 提供 `SubjectResolver` bean（`@FunctionalInterface`，返回当前主体，**必须是 Casdoor `sub`**）。
- `RemoteAuthzEngine` 实现全部 9 操作（端点映射见 §3 表），响应严格校验见 §2。

## 6. Casdoor → SpiceDB 身份同步

全部由 `authz.casdoor.enabled` 门控（**默认关**）。配置前缀 `authz.casdoor.*`：

| 配置项 | 默认 | 说明 |
|---|---|---|
| `enabled` | false | 总开关（装配 CasdoorClient/GroupSyncService） |
| `base-url` / `client-id` / `client-secret` | :8000 / 空 / 空 | 拉 Casdoor 数据 |
| `organization` | `built-in` | 单 org（`organizations` 为空时生效） |
| `organizations` | `[]` | **多 org 列表**；非空则逐 org 拉取合并（id 带 org 前缀无碰撞）。熔断仍全局 |
| `subject-field` | `id` | subject 取 `id`（=OIDC sub，推荐）或 `name` |
| `reconcile-enabled` / `reconcile-interval-ms` | false / 300000 | 定时对账（fixedDelay，**首跑在启动后一个间隔**） |
| `department-sync-enabled` | false | 部门树同步开关 |
| `delete-threshold` | 1000 | 单轮 DELETE 熔断上限，超则整轮放弃（含 TOUCH）；`<0`=不限 |

- **`GroupSyncService`（RBAC 组成员）**：期望态 Casdoor `groupMembers()` vs 当前态 `readRelationships` 读 **direct** `group#member@user` 元组（不误删嵌套组间接成员）→ 差量 TOUCH/DELETE，幂等。触发：`POST /admin/casdoor/sync`、webhook、reconcile。
- **`DepartmentSyncService`（部门树）**：Casdoor 嵌套 group（parentId→parent、role `<dept>-admin`→admin）→ SpiceDB `department` 的 `member/admin/parent` 三类元组；parent 单值，多余父边删除。触发：`POST /admin/casdoor/sync-departments`、webhook 联动、reconcile 定时器。
- **租户前缀**：组/部门 id 一律经 `CasdoorGroupIds.encode(org, group)` = `<org>_<group>`（org/group 仅 `[A-Za-z0-9-]`，含 `_` 即拒——防跨租户同名组串权、防分隔歧义）。

## 7. 授权模型（`.zed` schema）

两份 schema 由 deploy 脚本 `cat knowledge.zed his.zed | POST /v1/schema/write` **合并加载**（应用层只读不写 schema）。共 10 个 definition。

- **knowledge.zed（7）**：`user` / `group`（可嵌套）/ `organization` / `space`（`manage/edit/comment/view`，含 `parent_org->administrate`，`public_viewer: user:*` 公开链接）/ `folder`（沿 `parent_folder/parent_space` **向下继承**）/ `department` / `document`。
- **`document` 当前为部门层级隔离模型**（取代旧 D3）：`view = owner + viewer + public + home_dept->doc_reader`、`share = owner + home_dept->member + home_dept->doc_admin`、`edit = owner`；`department.doc_reader = member + parent->doc_reader` **沿部门树向上传播**（祖先看后代，方向与 space/folder 相反）。旧 `parent_space/parent_folder/editor/public_viewer` 仅兼容保留、不进 `view/edit`。权威文档：[authz-department-model.md](./authz-department-model.md)。
- **his.zed（3，数据权限样板）**：`dept`/`patient`/`encounter`——本科室 + 科主任 + 主治跨科 + 作者。
- **recsys.zed（4，广告主作用域模型；⚠️ 不参与上面合并，按 §B 写入 recsys 专属实例 :8544）**：`platform`（administrate/review/view_reports 平台职能锚点）+ `advertiser`（`manage = owner + platform->administrate`、`edit = member + manage`、`view = edit + platform->view_reports`）；广告/竞价词/创意不进 SpiceDB（权限纯继承，消费方反查 `advertiser_id` 再判）。fixture：`deploy/recsys-authz-fixture.sh`。
- **RBAC 是特例**：角色 = 一个 group + 一条边（绑到资源的对应 relation）。
- **多租户**：对象 id `<tenantId>_<id>`；**每项目独立 SpiceDB 实例**（§B 决策）——因此 `schema/write` 整体替换语义在项目内是安全的。

## 8. 一致性（ZedToken 三档）

| 档位 | 用途 |
|---|---|
| `MINIMIZE_LATENCY` | 默认判权，最快，允许轻微陈旧 |
| `AT_LEAST_AS_FRESH` | 写后读/撤权立即生效——带上写返回的 ZedToken |
| `FULLY_CONSISTENT` | 管理/调试；admin 读一律用它；core `readRelationships` 强制用它 |

坑：刚写入就读必须带 ZedToken 或 full，否则可能被量化快照漏读。server 的 ZedToken 水位缓存（见 §3）
兜底了"无 token 的 at_least_as_fresh"这一类；跨实例/跨服务写后读仍须调用方自带 token。

# 候选方案对比、风险评审与评分（risk-reviewer + plan-judge）

## 1. 统一比较前提

三个方案均必须：保留 7 个业务路由、`src/api/authz.ts` 接口、`src/auth/*` 鉴权、`authStore`、React Query、lexicon 和 Ant Design；不修改数据库或后端；不引入重型 UI 库。

## 2. 方案摘要

| 方案 | 核心差异 | 结构改动 | 产品能力改动 | 粗略成本 |
|---|---|---:|---:|---:|
| A 渐进视觉整顿 | 页面不拆，theme + shell + CSS Grid + 状态原语 | 低 | 极低 | 6–9 人日 |
| B 任务型工作台 | 页面容器保留逻辑，表单/结果/列表拆成呈现面板 | 中 | 低 | 10–14 人日 |
| C 上下文权限工作台 | URL 上下文、跨页带参、统一 Inspector | 高 | 高 | 17–25 人日 |

人日仅为单人、环境齐备的规划估算，不是承诺；需由执行团队按测试环境和评审流程校准。

## 3. 风险评审

### 3.1 兼容性

**A：低风险。** 路由、组件 state 和 API 调用位置基本不动。主要风险是全局 CSS 与 Antd 内部结构耦合。限制自有类选择器、使用 ConfigProvider component token 可控制。

**B：中低风险。** 页面容器保留请求和 handler，但拆子组件时可能改变受控值、事件时机、loading/disabled 逻辑。应以 props 契约测试锁住 payload；不要把 mutation 移入子组件。

**C：中高风险。** 除 B 风险外，还涉及 URL 与 local state 双向初始化、浏览器前进后退、非法/旧参数、路径分享兼容。需要明确参数版本或至少容错策略。

### 3.2 事务与一致性

前端没有数据库事务；grant/revoke 是单次 POST，成功后 invalidate relationships。三方案都不能宣称获得跨请求事务。

- A/B 不改变现有一致性边界。应验证 mutation 成功时才显示成功和刷新，失败不清空表单。
- C 的“授予后去验证”可能让用户误以为两步是原子事务；必须明确“已写入”和“验证查询”是两个独立请求，并分别反馈。
- check 与 expand 现在是并行独立请求。所有方案都必须允许 check 成功、expand 失败的部分成功状态。

### 3.3 并发、竞态与重复操作

- `api/client.ts` 的 401 刷新已有单飞逻辑，重设计不得复制第二套刷新。
- Grants 共用一个 `revoke` mutation；列表行撤销与表单撤销同时触发时，现有 mutation 状态语义可能交叉。A 最可能继续使用全局 busy；B 可呈现更精细状态，但若要真正区分每行 pending 就属于逻辑改动，应另立任务。
- resourceId 边输入边触发 query，快速切换由 React Query 按 key 隔离；但网络量可能增加。三方案默认不改变行为，性能优化需单独验收。
- Playground 多次运行可能使较早响应晚到。React Query mutation 的 `data` 行为需用测试确认；B/C 因结果面板更复杂，更需要连续提交竞态测试。
- 主按钮 pending 时应禁用，防止用户重复触发；当前 Button `loading` 通常会阻止点击，但列表撤销没有行级 pending 表达。

### 3.4 幂等

- IdentitySync 页面声明后端幂等，前端仓库无法验证；UI 只能防重复点击，不能证明接口幂等。
- grant/revoke 的幂等语义未在前端证明。成功提示不能使用“重复操作无影响”之类文案。
- A/B 对幂等风险无新增；C 若提供跨页重放/分享链接，更容易让用户重复执行，链接必须只预填，不自动提交。

### 3.5 性能

**A：** 最少新增组件和 JS；风险主要是 CSS 规则膨胀、Schema 卡片/Tree/List 大数据仍一次渲染。

**B：** 组件拆分本身开销可忽略；如果 props 不稳定可能增加渲染，但当前规模很小。可按需 `memo`，不应预优化。Audit 仍是前端 200 条分页，现有负载可控。

**C：** URL 同步、Inspector 与跨页摘要增加渲染和测试复杂度。若每次输入更新 URL，会造成 history/重渲染压力；应仅提交时同步。

共同限制：现有 API 没有 cursor/total，不能通过 UI 虚构服务端分页。长 expand 树和长元组必须提供容器滚动。

### 3.6 安全与权限

- 前端 gate 不是安全边界，后端 401/403 才是。A/B/C 都必须保留错误反馈。
- 颜色和禁用态不能泄露更多后端数据；viewer 菜单策略待产品确认。
- 撤销二次确认可降低误操作，但它是交互变更，必须确认后实施。
- C 会把资源/主体 id 放入 URL、浏览器历史、代理日志和截图，安全风险显著高于 A/B。默认不应落地，除非完成敏感数据评审。
- `.zed`、元组、错误消息按 React 文本节点渲染，不要改用 `dangerouslySetInnerHTML`。
- 主题/样式不能移除 focus outline；自定义 focus 必须达到可见性要求。

### 3.7 数据迁移与配置

- A/B 无数据迁移、数据库变更、API 消息结构变更；只新增 theme/CSS 与组件。
- C 无服务端数据迁移，但产生新的 URL 客户端契约，属于“链接格式迁移”风险。
- 三方案默认不加环境变量、不改 Nginx/Vite/OIDC 配置。
- 若引入测试依赖，会修改 `package.json` 与 `pnpm-lock.yaml`，不影响运行时产物逻辑。

### 3.8 灰度、观测与回滚

仓库未发现现成 feature flag 或前端遥测 SDK，不能虚构百分比灰度与监控平台。

- **A** 可按 shell → 页面逐个合并；回滚单页最容易。
- **B** 建议先发布 theme/shell/common primitives，再逐页替换；页面容器保留旧逻辑，单路由可 revert。
- **C** 必须同时考虑分享链接；代码回滚后新 URL 可能失效，回滚成本最高。
- 可用现有 HTTP/浏览器基础观测：静态资源错误、`/admin/*` 4xx/5xx、前端控制台错误、关键流程人工 smoke。具体日志/指标平台待运维确认。
- 构建产物应保留前一镜像/tag；仓库当前没有发布编排文件可证明具体回滚命令。

## 4. 典型失败场景矩阵

| 场景 | A | B | C | 必测控制 |
|---|---:|---:|---:|---|
| 侧栏折叠/移动 Drawer 后路由选中错误 | 中 | 中 | 中 | 7 路由逐一导航 |
| grant payload 丢 `subjectRelation` | 低 | 中 | 中 | group/organization fixture |
| check 成功但 expand 失败被显示成整体失败 | 中 | 低（分面板后） | 低 | 部分成功测试 |

| 全局 CSS 污染 Antd | 中高 | 低（CSS Modules） | 低 | 组件截图 + 禁止深层覆盖 |
| 旧结果跨 mode 混显 | 中 | 中 | 高（URL 状态） | 三模式切换回归 |

| 回滚后深链失效 | 无新增 | 无新增 | 高 | URL 兼容策略 |

## 5. 评分方法

评分 1–5，**5 表示最优/最容易控制**。对于“复杂度、测试难度、回滚成本”，5 分分别表示复杂度低、测试容易、回滚成本低。权重事先固定，避免为了偏好方案调整权重。

| 维度 | 权重 | A | B | C | 评分依据 |
|---|---:|---:|---:|---:|---|
| 正确性 | 25% | 4.2 | 4.6 | 3.6 | B 可用面板契约测试锁行为；C 新状态语义最多 |
| 改动风险 | 20% | 4.7 | 4.0 | 2.7 | A 最少触碰结构；C 涉路由上下文 |
| 复杂度 | 10% | 4.8 | 3.7 | 2.3 | 文件、状态与契约规模 |
| 可维护性 | 15% | 3.0 | 4.6 | 4.3 | A 保留大组件；B 边界最贴当前规模 |
| 扩展性 | 10% | 2.8 | 4.4 | 4.9 | C 对跨页排障能力最强 |
| 测试难度 | 10% | 4.0 | 3.8 | 2.2 | B 测试面增多但隔离更好；C 状态组合爆炸 |
| 回滚成本 | 10% | 4.8 | 4.0 | 2.5 | C 有已分享链接兼容负担 |
| **加权总分** | **100%** | **4.08** | **4.23** | **3.28** | 四舍五入到两位 |

## 6. 确认偏差检查

- B 分数最高不是因为“模块化必然更好”，而是核心两页确有持续增强价值，且页面容器可保留真实调用链。
- A 与 B 只差 0.15；如果交付窗口不足两周或测试基础设施无法及时建立，A 可能是更正确的选择。
- C 的产品体验上限最高，但当前任务明确是布局/视觉重设计，且仓库没有 relation→permission 映射或 URL 安全约束，因此本期选择 C 会过度扩 scope。
- 评分没有纳入主观“新颖度”；也没有因为核心页重要就默认接受最大改造。

## 7. 评审结论

不机械照搬单一方案：采用 **B 的页面容器/呈现面板边界和任务型布局**，吸收 **A 的分阶段、低风险上线方式与不改请求时机原则**；仅吸收 C 的“结果输入摘要”和未来“授予后验证”扩展位，不在本期引入 URL 状态、Inspector 或 relation→permission 自动映射。

该合并方案的已知弱点：文件数增加；需要先设计 props；跨页上下文仍不能恢复；在没有现成测试框架的仓库中会增加一次性基础设施成本。

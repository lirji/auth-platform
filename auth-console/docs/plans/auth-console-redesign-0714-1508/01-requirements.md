# 01 · 需求分析（requirements-analyst）

## 1. 分析口径与证据边界

本文只规划 `auth-console` 的页面布局与视觉重设计。结论来自当前仓库的 `src/pages/*`、`src/components/*`、`src/domain/*`、`src/api/*`、`src/auth/*`、`src/store/*`、`src/router/routes.tsx`、`src/nav.tsx`、`src/main.tsx`、`src/styles/global.css`、`package.json` 与 `README.md`。未读取后端实现或生产数据；涉及后端语义而前端代码无法证明的内容均标为“待验证”。

## 2. 产品定位与用户

- 产品是统一权限平台的**授权侧管控台**，核心对象是 SpiceDB 关系、授予、判定与反查。
- 主要用户是平台管理员、安全管理员；`authStore.ts` 识别 `authz-admin` 与 `authz-viewer`，后者只读。
- 身份、组织、用户、组的增删属于 Casdoor 控制台，本项目不复制该能力。
- 核心价值页是 `/grants` 授予管理和 `/playground` 权限调试器；信息架构、首屏入口和视觉权重都应优先突出它们。

## 3. 已确认业务规则

1. relation 与 permission 是两套词汇：relation 用于 grant/revoke，permission 用于 check/lookup。证据：`domain/lexicon.ts`、`components/domain/selects.tsx`、`api/authz.ts`。
2. 授予元组格式为 `resourceType:resourceId#relation@subjectType:subjectId[#subjectRelation]`；`TupleText` 负责等宽展示。
3. `GrantsPage` 的主体类型当前限定为 `user | group | organization`；资源类型排除 `user`。
4. `group` 或 `organization` 主体在 userset 开启时带 `#member`。当前代码提供开关，但文案称“必须开”；这存在交互与业务约束不一致，需作为待验证项处理，不能擅自改成强制逻辑。
5. 授予或撤销成功后，失效当前 `['relationships', resourceType, resourceId]` 查询并刷新现存授予列表。
6. 现存授予列表中的撤销会原样携带 `subject.relation`，避免 group userset 撤销丢失 `#member`。
7. 调试器有三种模式：直接判定、按主体反查资源、按资源反查主体。
8. 直接判定要求主体 id、资源 id、permission；同时 best-effort 调用 expand。check 结果是权威允许/拒绝显示，expand 只用于解释路径。
9. 反查主体结果当前明确提示 caveated 命中未过滤，需结合直接判定核对；重设计不能弱化这一风险提示。
10. SchemaViewer 从 `GET /admin/schema` 读取 `.zed`，用 `parseZed` 做轻量解析，可切换原始文本。
11. IdentitySync 从 Casdoor 拉取组成员，差量写入/移除 `group#member`；页面文案声明幂等，实际后端幂等性未在本仓库验证。
12. Audit 请求最近 200 条，表格前端每页 20 条；页面声明后端为内存环形最近 500 条且重启清空，此描述仅来自当前 UI，待与后端确认。
13. 所有业务路由在 `ProtectedRoute` 下，要求 `authz-viewer` 或 `authz-admin`；当前路由和菜单没有逐页 admin gate。
14. API access token、OIDC 登录、React Query 缓存、Zustand 鉴权镜像均保留，不纳入重写。

## 4. 信息架构需求

### 4.1 一级导航

保留 7 条现有路由和路径，避免深链、书签与运维入口失效：

| 分组 | 页面 | 路径 | 导航优先级 |
|---|---|---:|---|
| 工作台 | 概览 | `/` | 中 |
| 核心操作 | 授予管理 | `/grants` | 最高 |
| 核心操作 | 权限调试器 | `/playground` | 最高 |
| 资源与模型 | 授权模型 | `/schema` | 高 |
| 资源与模型 | 空间/知识库 | `/spaces` | 低，当前占位 |
| 系统治理 | 身份同步 | `/sync` | 中 |
| 系统治理 | 审计日志 | `/audit` | 高 |

侧栏可通过拟扩展 `NavItem` 增加分组/说明/访问级别元数据，但不改现有 path。当前 `NAV` 注释称其为菜单与路由单一配置源，实际上 `routes.tsx` 仍单独声明路由；重设计不顺带重构路由架构。

### 4.2 页面框架

- 固定/吸顶壳层：桌面侧栏 224px（折叠 72px）、顶栏 56px；移动端侧栏改 Drawer，不挤压主内容。
- 顶栏左侧展示折叠按钮与面包屑，右侧保留账户菜单；当前页标题下沉到内容区的 `PageHeader`，避免 Header 同时承担全局与页面标题。
- 内容区使用浅中性底色、最大宽度 1440px、居中，桌面外边距 24px；核心工作台允许使用全宽上限。
- 页面统一采用“标题/说明/主要动作 → 页面主体 → 辅助说明”的垂直节奏。
- 面包屑最多两级（分组 / 页面），当前没有详情子路由，不虚构层级。

### 4.3 关键流程

**授予：**选模式 → 选资源类型与输入 id → 选 relation → 选主体类型与输入 id → 确认 userset/元组预览 → 提交 → 就地反馈 → 现存授予刷新。

**调试：**选模式 → 输入该模式必要条件 → 运行 → 显示 loading skeleton → 展示 ALLOW/DENY、反查列表或空态 → check 模式独立展示 expand 路径失败/成功状态。

**模型理解：**类型卡片/原始 `.zed` 切换 → 识别对象 → 分区查看 relation 与 permission → 必要时阅读表达式。

**审计排查：**查看最近 200 条 → 分页 → 手动刷新。筛选、搜索、导出在现有 API 中不存在，本期不得假定可实现。

## 5. UI 设计系统需求

### 5.1 设计原则

- 管理台优先清晰、可扫描、可追责，不使用大面积渐变、玻璃拟态或装饰性插画。
- 色彩不作为唯一状态信号；ALLOW/DENY、relation/permission 同时使用文字、图形/前缀、边框样式区分。
- 保留 `lexicon.ts` 对具体对象、关系、权限的中文标签与颜色；新增的是类别级视觉语法，不另建冲突映射。
- ID、元组、`.zed` 表达式使用等宽字体；普通说明与表格正文使用系统无衬线字体。

### 5.2 建议 token（由 `ConfigProvider theme` 驱动）

| 类别 | 建议值 | 用途 |
|---|---|---|
| `colorPrimary` | `#315EFB` | 主动作、当前导航、焦点 |
| `colorInfo` | `#315EFB` | 信息反馈 |
| `colorSuccess` | `#16A36A` | ALLOW、成功 |
| `colorWarning` | `#D97706` | caveat、待确认 |
| `colorError` | `#D92D20` | DENY、错误、危险撤销 |
| `colorBgLayout` | `#F5F7FA` | 应用背景 |
| `colorBgContainer` | `#FFFFFF` | 卡片/表格/面板 |
| `colorText` | `#172033` | 主文本 |
| `colorTextSecondary` | `#667085` | 次文本 |
| `colorBorderSecondary` | `#E6EAF0` | 卡片/分割线 |
| `borderRadius` | `8` | 输入、按钮、基础容器 |
| `borderRadiusLG` | `12` | 页面卡片、结果面板 |
| `fontSize` | `14` | 正文 |
| `fontSizeHeading1` | `28` | 页面标题（建议最终 CSS 控制 24–28） |
| `controlHeight` | `36` | 默认控件 |
| `controlHeightLG` | `40` | 核心表单 |
| `boxShadowSecondary` | 轻阴影 | 浮层；普通卡片优先边框而非阴影 |

中性阶建议在 CSS 变量中补齐：`#172033 / #344054 / #667085 / #98A2B3 / #D0D5DD / #E6EAF0 / #F2F4F7 / #F5F7FA / #FFFFFF`。间距采用 4px 基线：4、8、12、16、24、32、40、48。

### 5.3 relation 与 permission

- relation：前缀 `R` 或连接图标、描边 Tag、标题固定为“关系（可授予）”。
- permission：前缀 `P` 或盾牌图标、浅色填充 Tag、标题固定为“权限（可判定）”。
- 具体 Tag 的色相仍来自 `RELATIONS` / `PERMISSIONS`；类别差异不能仅靠蓝/绿两色。
- 表单只在授予页出现 RelationSelect；调试器只出现 PermissionSelect。

### 5.4 状态反馈

| 状态 | 统一规范 |
|---|---|
| 首次加载 | 页面骨架 Skeleton；避免仅用居中 Spin 造成布局跳变 |
| 局部刷新 | 保留旧内容并在面板标题/刷新按钮显示 spinning |
| 未执行 | 使用带操作提示的 Empty，不与“查询结果为空”混淆 |
| 查询为空 | Empty + 当前查询条件摘要；不显示成错误 |
| 页面级错误 | Alert/Result + 人话错误 + 重试按钮 |
| 写入成功 | `message.success` + 面板刷新状态；token 作为次要追踪信息 |
| 危险操作 | 撤销使用 danger，列表内快速撤销增加 Popconfirm（视觉方案建议，是否增加确认需产品确认） |
| 权限不足 | 控件禁用/只读提示只改善体验；后端 403 仍是安全边界 |
| expand 失败 | 不覆盖 check 结果；判定路径面板单独提示“解释不可用” |

## 6. 页面级骨架需求

### 概览

- 第一屏用两个明显的快捷任务卡突出“授予管理”和“权限调试器”，授权模型为次级入口。
- 当前身份改为紧凑侧卡/描述列表；viewer 提示保留。
- “暂无规模统计”作为弱提示，不伪造统计 KPI。

### 授予管理

- 桌面采用 7:5 双栏：左侧流程表单，右侧现存授予；右侧可 sticky，但不能遮挡页面底部。
- 表单分“资源 / 关系 / 主体 / 确认”四段；元组预览始终可见，缺失段用 `?` 保留当前语义。
- grant/revoke 模式视觉清晰，但 revoke 不能用整页红色；仅主按钮、警示和确认态使用危险色。
- 现存授予要区分未输入、加载、错误、为空、有数据五态；当前 `Card loading` + `List emptyText` 不足。
- viewer 进入页面时的只读行为待确认；不得只凭前端隐藏形成安全假象。

### 权限调试器

- 桌面采用 5:7 双栏，右侧结果获得更大空间；结果区至少 360px 高，防止运行前后跳变。
- 三模式切换保留；必要输入随模式变化，保留“用我自己”。
- check 结果顶部为紧凑但高辨识度的 ALLOW/DENY banner，下面是“本次输入摘要”和“判定路径”。
- expand 与 check 独立反馈；路径树可能很宽，需要横向滚动和长 ID 换行策略。
- 反查列表显示总数（仅当前响应长度，不能称服务端总数），结果为空与尚未运行分开。
- caveat 警示必须在反查主体结果附近持续可见。

### 授权模型

- 顶部切换“类型卡片 / 原始 `.zed`”；卡片使用响应式网格。
- 卡片标题显示对象类型 Tag 与中文名；内部 relation、permission 两区有固定图例。
- 无解析结果不能显示空白，应提示“未解析出 definition，可查看原始文本”；`parseZed` 不是完整解析器。
- 原始文本放入可滚动代码面板；复制按钮属于可选增强，不是必需。

### 审计

- 使用无外层重 Card 的页面表格容器，工具条包含标题、范围说明与刷新。
- 保留现有四列与 pageSize 20；`detail` 需换行/省略兼顾，完整值可通过 tooltip/copy（待选择）。
- 当前 API 无筛选、排序、服务端分页契约，不能在方案中伪造这些能力。
- 保留“内存环形、重启清空”的数据持久性提示，但上线前需后端确认文案。

### 身份同步

- 单任务页使用 640–720px 的聚焦面板；说明、主动作、最近一次本地响应结果分区。
- loading 时防重复触发；错误显示可重试。页面刷新后没有同步历史，不能称“最近一次同步记录”。

### 空间管理

- 保持占位事实，用正式 Empty/Result 呈现“暂未开放”，避免伪装成已实现页面。
- 是否从一级导航隐藏或标记“规划中”待确认；本期不新增空间 CRUD。

## 7. 响应式与信息密度

- `≥1200px`：核心页双栏；内容最大宽 1440px。
- `992–1199px`：侧栏可折叠，双栏比例收紧，表格允许横向滚动。
- `768–991px`：侧栏 Drawer；核心页单栏，结果区位于输入区下方。
- `<768px`：页面边距 16px，Compact 组合在必要时改为上下排列，按钮主操作全宽；审计表保留关键列并横向滚动，不擅自删字段。
- 默认密度为“舒适管理台”（36/40px 控件、表格 middle）；Audit 可用 small/middle，但全站不提供密度切换，除非用户确认需要。

## 8. 非目标

- 不修改业务逻辑、API 路径/请求响应结构、OIDC 流程、权限 claim、React Query/Zustand 选型。
- 不修改数据库，不新增后端接口，不做数据迁移。
- 不新增身份、组织、用户、组 CRUD。
- 不新增图形化 Schema 编辑器、关系图重型图库、全局搜索、审计导出或复杂筛选。
- 不实现暗色主题；token 应保留未来扩展空间，但本期只验收浅色主题。
- 不改变现有 7 个业务路由。

## 9. 歧义、待验证与易遗漏点

| 编号 | 问题 | 当前证据 | 规划处理 |
|---|---|---|---|
| Q1 | group/organization 的 `#member` 是否必须强制 | UI 有可关闭 Switch，文案称必须开 | 首期仅重排与强化提示；强制逻辑需业务确认 |
| Q2 | viewer 是否可见授予/同步菜单 | 路由和 NAV 全量可见，后端写接口应返回 403 | 在未确认前保持现有可见性与请求行为，只增加角色说明；是否 gate/禁用另行决策 |
| Q3 | Audit“最近 500 条内存环形”是否仍准确 | 仅页面文案 | 上线前接口负责人确认，不准确则改文案 |
| Q4 | expand 响应是否稳定符合 `SpiceNode` | 前端以可选字段解析 `unknown` | 保持容错；增加契约 fixture 测试，真实结构待联调 |
| Q5 | 空间管理应保留导航还是隐藏 | 当前是占位路由 | 默认保留并标“规划中”，最终由产品确认 |
| Q6 | 是否允许增加撤销二次确认 | 当前表单撤销和列表撤销均直接请求 | 属安全 UX，建议增加；纳入实施前确认点 |
| Q7 | 长 ID、中文/英文混排的典型长度 | 仓库无样本 | 用 8/32/128 字符与无空格字符串做视觉测试 |
| Q8 | 浏览器支持矩阵和无障碍等级 | 仓库未声明 | 默认最近两个版本 Chrome/Edge，WCAG 2.1 AA 作为建议，待确认 |
| Q9 | 是否需要品牌 Logo/名称规范 | 只有文字“权限管控台” | 不制作新品牌资产，使用文字标识 |

## 10. 验收标准

1. 现有 7 个路由、OIDC 登录/回调/登出及所有 API 调用路径保持不变。
2. 1440px 下壳层、面包屑、页面标题、内容网格对齐统一；核心两页分别按 7:5 与 5:7 双栏呈现。
3. 768px 以下无侧栏挤压、无关键操作不可达；表格和代码/树内容可滚动。
4. 所有页面明确覆盖 loading、未执行/empty、error、success；核心页区分未运行与空结果。
5. relation 与 permission 在标题、前缀/图标、Tag 样式上至少两种非颜色差异，同时保留 lexicon 的标签与色值来源。
6. ALLOW/DENY 不仅依赖颜色；expand 失败不抹去 check 结果。
7. 授予元组预览、group/organization userset、列表撤销携带 `subjectRelation` 的现有规则不回归。
8. Audit 不展示不存在的服务端筛选/总数/持久化能力。
9. `pnpm build` 通过；新增测试工具后，测试脚本通过；关键页面完成桌面/平板/手机视觉回归。
10. 最终 Git 变更仅包含批准的前端实现文件；本规划阶段则仅包含本目录文档。
